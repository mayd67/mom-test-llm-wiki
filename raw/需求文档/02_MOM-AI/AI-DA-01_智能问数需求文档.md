## 1. 业务背景与价值目标

### 1.1 业务问题与挑战

**挑战1：业务问题表达不确定，系统难以一次理解**
- 问题：同一指标存在口语别名、简称和多种问法，且常缺少时间窗口或组织范围。
- 影响：查询前反复沟通，会议现场无法快速进入数据判断。

**挑战2：用户问题经常缺失关键条件**
- 问题：问句仅有“查一下交付率/异常情况”，缺少时间、对象、对比维度等必要条件。
- 影响：系统直接查询会造成误解；人工补问效率低且结果不稳定。

**挑战3：结果展示诉求不固定**
- 问题：不同角色对展示偏好不同，有时需要趋势，有时需要对比、TopN或明细清单。
- 影响：固定报表难匹配临时问题，结果虽有数据但不可直接用于讨论。

**挑战4：查询结果缺少口径与边界说明**
- 问题：结果常缺少指标定义、样本范围、刷新时间和权限边界说明。
- 影响：结论可信度不足，复盘时容易产生口径争议。

### 1.2 价值主张与目标

| 价值主张 | 对应挑战 | 业务目标 |
| --- | --- | --- |
| 支持口语化、多表达方式问句理解 | 挑战1 | 提升问句识别成功率，降低人工翻译成本 |
| 对不确定问题做最小澄清后再查询 | 挑战2 | 缩短“提问到可用结果”路径，提升一次查询成功率 |
| 根据问题自动匹配展示方式并支持切换 | 挑战3 | 提升结果可读性和会议现场可用性 |
| 输出结果强制附带口径与边界说明 | 挑战4 | 提升可解释性，减少口径争议 |

### 1.5 术语及缩写解释

| 术语 | 说明 |
| --- | --- |
| 智能问数 | 用户以自然语言提问，系统完成条件澄清、查询执行和结果展示的能力 |
| 问题不确定性 | 问句缺失条件、表达模糊、别名混用、多口径并存等情况 |
| 澄清问题 | 系统针对缺失条件提出的最小必要补充问题 |
| 口径回显 | 在结果里直接说明“怎么算、算哪个时间段、算哪个组织范围”（例如：准时交付率公式+时间窗口+工厂范围）。 |
| 权限边界控制 | 只返回当前角色有权限看的数据（例如：操作工只能看本人任务相关数据）。 |
| 多样展示 | 根据问题选择表格、趋势、对比、TopN、明细等展示方式 |

## 2. 业务场景定义（角色 × 任务 × 问题）

### 2.1 场景建模原则

- 面向系统全角色建设问数能力；首批用系统初始默认16个角色做测试基线。
- 场景表达以“谁在什么任务下问什么问题”为主，不按模块功能堆砌。
- SIPOC用于统一边界定义，只做一份全局边界，不对每个场景重复建模。
- 场景七步法只用于高价值、高频、高风险场景做详卡，其他场景用轻量场景卡即可。
- 每个场景必须能映射到功能点、验收条目、测试问句，保证可验证。

### 2.2 全局问数链路（统一SIPOC）

| SIPOC要素 | 通俗定义 | 本需求中的约定 |
| --- | --- | --- |
| 供应方（S） | 数据和规则由谁提供 | 业务系统、指标口径定义、权限规则、角色上下文 |
| 输入（I） | 问数需要什么输入 | 用户自然语言问题、时间范围、组织范围、对象范围、用户身份 |
| 过程（P） | 系统怎么完成一次问数 | 意图识别 -> 条件补齐/澄清 -> 查询执行 -> 结果解释 -> 展示推荐 -> 追问下钻 |
| 输出（O） | 用户最终看到什么 | 结果摘要、口径说明、表格/图表/TopN/明细、可继续追问入口 |
| 客户（C） | 谁使用结果 | 系统所有角色（按权限返回）；首批16角色作为验收基线 |

## 3. 业务需求方案（非技术）

### 3.1 目标业务流程（To-Be）

1. 用户输入自然语言问题。  
2. 系统识别意图并判断是否存在关键条件缺失。  
3. 若仅存在低风险语义缺省，则允许默认补全并在结果回显中显式说明；若缺失或歧义会明显改变结果，则发起最小澄清。  
4. 系统执行查询并输出结果摘要与口径回显。  
5. 系统推荐展示方式并支持切换。  
6. 用户连续追问，系统继承上下文完成下钻。  
7. 系统输出对象TopN或明细列表，并保留失败分流提示。

```mermaid
    A[用户输入问题] --> B[意图识别]
    B --> C{条件是否完整}
    C -- 否 --> D[最小澄清]
    D --> E[条件确认]
    C -- 是 --> E
    E --> F[执行查询]
    F --> G[结果与口径回显]
    G --> H[多样展示与切换]
    H --> I[连续追问与下钻]
    I --> J[对象TopN/明细输出]
```

### 3.2 功能清单

| 功能编号 | 功能点 | 用户故事（EARS） | 关键业务规则 | 验收标准 | 优先级 |
| --- | --- | --- | --- | --- | --- |
| F-01 | 问句理解与意图识别 | 作为生产厂长/生产主管/班组长，我希望系统能理解口语化问句，以便我不用按固定术语提问也能查到数据。 | 识别后需回显系统理解的查询意图。 | 可识别同义问法并回显理解结果。 | P1-MVP |
| F-02 | 不确定条件澄清 | 作为任一业务角色，我希望系统在条件缺失时先问关键问题，以便避免查出错误口径。 | 仅澄清最小必要条件（时间、组织、对象、口径）。 | 澄清后可一次执行查询并得到结果。 | P1-MVP |
| F-03 | 查询生成与口径回显 | 作为各层级角色，我希望结果里能看到指标口径和范围，以便判断结果是否可信。 | 查询结果必须附带口径、时间窗口、组织范围说明。 | 所有结果可见口径回显，不允许“黑盒结果”。 | P1-MVP |
| F-04 | 多样展示选择 | 作为管理层与执行层角色，我希望同一结果可切换趋势/对比/TopN/明细，以便快速讨论或执行。 | 至少支持表格、趋势、对比、TopN、明细五类展示。 | 用户可在会话内切换展示方式。 | P1 |
| F-05 | 连续追问与上下文继承 | 作为任一角色，我希望基于上轮结果继续追问，以便减少重复输入。 | 同一会话追问默认继承上轮时间和组织条件。 | 多轮追问保持上下文一致。 | P1-MVP |
| F-06 | 失败分流与降级提示 | 作为任一角色，我希望遇到无数据/冲突口径/权限不足时有明确提示，以便快速决定下一步。 | 失败类型必须区分并给出可执行建议。 | 失败场景可定位原因并给出处理路径。 | P1-MVP |

### 3.3 关键业务规则

- 问句含关键条件缺失时，必须先澄清再查询。
- 澄清遵循“最小必要”原则，避免过度追问。
- 对低风险语义缺省允许默认补全，但必须在结果回显中显式说明补全假设。
- 若对象类型歧义会显著改变结果、权限边界或链路结构，则必须先最小澄清，不得静默补全。
- 查询结果必须回显：指标定义、时间窗口、组织范围、样本说明。
- 结果默认给出一种最匹配展示，并允许用户切换展示方式。
- 连续追问默认继承上下文，用户可显式覆盖。
- 所有查询和下钻均受组织、对象、字段权限限制。

### 3.4 数据口径与查询对象边界

#### 3.4.1 指标口径最小要求

| 口径项 | 要求 |
| --- | --- |
| 指标定义 | 必须有业务定义，避免同名异义 |
| 计算公式 | 必须可展示分子/分母或构成项 |
| 时间窗口 | 必须可见并可修改 |
| 组织范围 | 必须可见并可修改 |
| 样本说明 | 必须显示样本量或覆盖范围 |
| 刷新时间 | 必须显示最近刷新时间 |

#### 3.4.2 查询对象边界

| 对象类型 | 示例 | 要求 |
| --- | --- | --- |
| 核心对象 | 生产订单、制造订单、异常任务 | 支持下钻到对象清单 |
| 维度对象 | 工厂、车间、产线、产品、工序 | 支持筛选、对比、贡献分析 |
| 权限对象 | 组织、用户、角色 | 用于权限过滤与边界控制 |

补充规则：

- 在定制追溯类问题的 V1 中，用户泛称“订单”默认按“生产订单”理解。
- 用户显式说“制造订单”或对象编号可稳定识别对象类型时，按实际对象类型处理。
- 若“订单”可能同时指向生产订单与制造订单，且会明显改变结果，则先最小澄清。

### 3.5 非功能要求

| 类别 | 要求 |
| --- | --- |
| 性能 | 常规查询 P95 <= 3秒；多维展示切换 P95 <= 5秒 |
| 可用性 | 查询失败可重试和条件修正，不中断会话 |
| 可解释性 | 每次输出必须包含口径回显与范围说明 |
| 可追溯性 | 会话、条件、结果、追问链路可追踪 |
| 安全性 | 结果和下钻必须遵守组织/对象/字段权限 |

## 4. 问数编排方案（Agent/Skill层）

### 4.1 编排目标

- 把“自然语言问题”转换为“可执行查询条件”。
- 把“不确定输入”转换为“最小澄清路径”。
- 把“查询结果”转换为“可解释、可展示、可追问”的输出。

### 4.2 编排链路

1. 意图识别：识别指标、对象、时间、组织、展示诉求。  
2. 缺项检测：判断是否缺失关键条件。  
3. 最小澄清：仅对缺项提问。  
4. 查询执行：执行后返回结构化结果。  
5. 展示编排：按问题和角色推荐展示方式。  
6. 追问下钻：继承上下文并定位到对象明细。  
7. 失败分流：无数据、口径冲突、权限不足分别处理。

### 4.3 人机协同分工

| 环节 | AI职责 | 用户职责 |
| --- | --- | --- |
| 问题理解 | 解析问句并回显理解结果 | 确认或修正理解结果 |
| 条件补齐 | 发起最小澄清并补齐条件 | 回答澄清问题 |
| 结果展示 | 提供默认展示并支持切换 | 选择更适合的展示视图 |
| 追问下钻 | 继承上下文并定位对象 | 选择继续追问方向 |
| 异常处理 | 识别失败原因并给建议 | 决定是否调整条件重查 |

### 4.4 场景任务到Skill映射

| 场景任务 | Skill | 输入契约 | 输出契约 | 失败处理 |
| --- | --- | --- | --- | --- |
| 标准查询 / 指标查询 | `intelligent-query` | 问句、时间窗、组织范围（可缺失） | 条件澄清 + 查询结果 | 缺项时先澄清 |
| 查后分析 / 归因 / 贡献 / 建议 | `intelligent-analysis` | 上游结果、问题原文、时间窗、组织范围、口径上下文 | 分析5段结果 | 缺少分析基础时先提示补查 |
| 单对象过程追溯 / 质量履历 / 异常链路 | `production-order-trace-query` | 锚点对象、追溯焦点、时间窗（可缺失） | 追溯5段结果 | 多候选对象或链路缺失时显式提示 |
| 失败分流提示 | `Agent` | 查询失败上下文 | 失败类型 + 修正建议 | 权限不足不返回明细 |

补充说明：

1. V1 允许 Agent 按问题清单或规则显式路由，不要求自动编排。
2. 复合问题按“先查后析”或“先追溯后析”串行处理。
3. 详细规则见 `AI-DA-01_Agent路由规则草案.md`

### 4.5 输出结构（问数统一5段）

1. 查询理解与条件回显  
2. 结果摘要  
3. 展示结果（表格/趋势/对比/TopN/明细）  
4. 可继续追问建议  
5. 口径与限制说明

补充说明：

- 以上 5 段适用于查询、分析、定制追溯三类 Skill。
- 定制追溯类第 5 段可表述为“来源与限制说明”，但仍需包含已覆盖对象、缺失链路、权限限制与数据更新时间。

### 4.6 异常与降级策略

- 无数据：提示放宽条件（时间/范围/对象）。  
- 权限不足：返回权限限制说明，不返回越权数据。  
- 口径冲突：展示可选口径并要求确认。  
- 样本不足：提示样本不足及风险，不输出确定结论。
- 锚点对象未找到：提示检查对象编号、时间范围或组织范围。
- 命中多个候选对象：列出候选并要求确认。
- 追溯链路部分缺失：返回已确认链路，并明确缺口。
- 缺少分析基础结果或比较基线：先提示补充查询或确认基线。

## 附录B：Skill执行细则（`data-analysis-agent`）

> 补充说明：V1 对外仍可保持统一问数入口；对内可按 `intelligent-query`、`intelligent-analysis`、`production-order-trace-query` 分层执行。本附录保留统一5段输出约束，路由细则见 `AI-DA-01_Agent路由规则草案.md`

### B.1 输入契约

- 用户问句（允许口语化和条件缺失）
- 指标标识（`kpi_id`或指标名称，可由问句解析）
- 时间窗口（本周/本月/自定义）
- 组织范围（工厂/车间/班组/个人）
- 维度上下文（产品、工序、设备、供应商等）

### B.2 输出契约（问数5段）

1. 查询理解与条件回显  
2. 结果摘要  
3. 展示结果（表格/趋势/对比/TopN/明细）  
4. 可继续追问建议  
5. 口径与限制说明

### B.3 执行流程要点

1. 先理解问句并识别缺项。  
2. 对低风险语义缺省允许默认补全并显式回显；高歧义场景先最小澄清。  
3. 生成查询并执行。  
4. 返回结果并推荐展示方式。  
5. 支持追问下钻和失败分流。

### B.4 异常与降级

- 无数据：返回放宽条件建议。  
- 权限不足：返回限制说明，不泄露明细。  
- 口径冲突：给出口径选项并要求确认。  
- 样本不足：提示补数路径与置信度限制。
- 锚点对象未找到：提示检查对象编号、范围或对象类型。
- 命中多个候选对象：列出候选供确认。
- 追溯链路缺失：显式说明已覆盖链路与缺失链路。
