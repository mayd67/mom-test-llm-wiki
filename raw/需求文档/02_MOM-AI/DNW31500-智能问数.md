# AI-DA-01 产品需求文档：自然语言数据查询与多样展示

> 版本：v0.4  
> 状态：草案（精简重排）  
> 日期：2026-03-17  
> 范围：仅覆盖核心场景1（智能问数）

## 1. 建设背景与价值目标

### 1.1 建设背景

为落实 MOM+AI 场景建设，本期新增“自然语言数据查询与多样展示”能力（AI-DA-01）。  
该能力支持用户用自然语言发起查询，在条件不完整时进行最小澄清，并输出可解释、可追问、可下钻的结果。

本能力长期目标适用于系统所有角色；首批以计划员相关角色为试点范围开展验收。

| 建设动因 | 本期新增能力 | 业务价值 |
| --- | --- | --- |
| 建立统一问数入口 | 自然语言查询 + 最小澄清 + 多样展示 | 降低查询门槛，提升查询效率 |
| 支撑不确定问题查询 | 条件补齐 + 计算说明 + 连续追问 | 提升结果可解释性与决策可用性 |
| 保障结果可用与可控 | 权限控制 + 试点验收基线 | 保证可用性与安全性并行 |

### 1.2 价值目标

1. 让用户用自然语言完成动态查数（含 KPI 类查询）。
2. 把“不确定问题”转成“可执行查询”。
3. 输出“可解释、可追问、可下钻”的结果，减少二次沟通成本。
4. 让结果能够说明数据从哪里来、按什么范围统计、当前有哪些限制，便于业务判断是否可直接使用。

## 2. 术语澄清与范围边界

### 2.1 术语澄清

| 术语 | 定义 | 本文使用说明 |
| --- | --- | --- |
| Agent | 会话与任务编排单元 | 负责问题理解、最小澄清、任务分发、上下文管理、结果整合与回显 |
| Skill | 被调用的能力单元 | 负责在给定输入下完成查询或分析并返回标准结果 |
| 智能问数 | 根据自然语言问题生成数据结果 | 重点回答“是多少/有哪些” |
| 数据分析 | 基于已有数据结果做解释与建议 | 重点回答“为什么/影响多大/下一步做什么” |
| 单轮问数查询 | 单轮、独立问题 | 不依赖上轮上下文 |
| 连续追问 | 多轮有关联问题 | 默认继承上轮时间、组织、对象等上下文，可显式覆盖 |
| 数据根因分析 | 基于现有数据做原因拆解 | 输出“可能原因与贡献”，不承诺绝对因果 |
| 计算说明 | 指标怎么算、统计范围是什么 | 每次结果必须展示 |
| 数据更新时间 | 结果对应数据刷新截止时点 | 每次结果必须展示 |
| 上游结果 | 在当前分析前已经查到的结果 | 用于支撑“先查后析” |
| 补充查询 | 为核实分析判断所做的少量补充查数 | 仅在确有必要时使用 |
| 数据来源 | 本次结果来自正式业务数据还是演示数据 | 每次结果应说明 |
| 时间口径 | 本次统计使用的时间范围与比较基线 | 每次结果应说明 |

说明：智能问数与数据分析可以组合，也可以独立。可“先查后析”，也可“只查不析”。

### 2.2 纳入范围

| 分类 | 纳入内容 | 示例 |
| --- | --- | --- |
| 查询能力 | 自然语言查询（含口语/别名） | 查询今天新进入的订单列表 |
| 指标能力 | 指标类查询纳入动态问数 | 本周达成率、合格率、废品率是多少 |
| 处理能力 | 条件不完整时最小澄清 | 缺时间范围时先问“本周还是本月” |
| 输出能力 | 展示计算说明 | 明确“怎么算、算哪段时间、算哪个组织范围” |
| 展示能力 | 多样展示 | 表格、趋势、对比、TopN、明细 |
| 分析能力 | 连续追问与根因分析 | 未达成和报废的主要原因有哪些 |
| 安全能力 | 权限控制 | 仅返回当前角色授权范围数据 |

### 2.3 排除范围

| 分类 | 不在本需求内 |
| --- | --- |
| 知识问答 | 制度解释、FAQ、文档检索、纯指标口径说明 |
| 首页能力 | 首页 KPI 卡片定义、编排、布局、跳转 |
| 协同边界 | 与 `ROLE-WB-AI-01 首页工作台` 保持独立；本需求仅复用统一计算说明与权限规则 |
| 高阶算法 | 预测、自动优化、自动排程等高级决策算法 |

## 4. 问题处理规则与输出要求

### 4.1 单轮问题处理规则（对应 S-01~S-13）

1. 问题信息完整时直接执行查询或诊断，不额外打断。
2. 优先返回与问题最匹配的展示方式（列表或图表）。
3. 支持从结果进入对象明细下钻。

### 4.2 连续追问处理规则（对应 C-01）

1. 默认继承上轮时间范围、组织范围、对象范围。
2. 用户显式指定新条件时，覆盖默认继承。
3. 链路中每一步都需给出可继续追问建议。
4. 对低风险语义缺省可做默认补全，但必须在“查询理解与条件回显”中显式说明补全假设。
5. 若对象类型歧义会明显改变结果、权限范围或链路结构，则必须先做最小澄清，不得静默补全。

### 4.3 输出最小要素

每次输出必须包含以下 5 项：
1. 查询理解与条件回显
2. 结果摘要
3. 展示结果
4. 可继续追问建议
5. 口径与说明（含数据来源、是否为演示数据、是否做过补充查询、时间口径是否一致、数据更新时间）

补充说明：
1. 以上 5 项同时适用于统一入口、查询能力、分析能力和追溯能力。
2. 追溯类问题的第 5 项可表述为“来源与限制说明”，但至少仍需包含追溯范围、已覆盖对象、缺失链路、数据更新时间与权限限制。

### 4.4 异常处理规则

需明确区分并提示以下异常：
1. 无数据
2. 权限不足
3. 计算说明冲突
4. 样本不足
5. 锚点对象未找到（适用于定制追溯类）
6. 命中多个候选对象需确认（适用于定制追溯类）
7. 追溯链路部分缺失或中断（适用于定制追溯类）
8. 缺少分析基础结果或比较基线（适用于分析类）
9. 当前证据不足，只能说明现象，不能直接判断原因

### 4.5 数据可信底线

1. 同一项目内同一指标必须使用唯一计算说明版本；不同项目可按项目级配置不同计算说明。
2. 时间范围、组织范围和比较基线必须可见且可修改。
3. 组织/对象/字段三层权限必须生效。
4. 数据刷新间隔支持配置；每次结果必须显示数据更新时间。
5. 数据不足或规则冲突时必须明确提示“当前限制”。
6. 核心指标结果应与基线报表一致；汇总结果应能与订单明细相互印证。
7. 每次结果都应说明来自正式数据还是演示数据。
8. V1不演示清单外问题，不纳入首批验收范围。

## 5. 统一入口与能力分工

### 5.1 拆分目的

1. 对用户保持一个统一入口，降低使用门槛。
2. 对内将查数、诊断分析、追溯分开建设，便于稳定交付。
3. 先保障首批清单问题可用，再逐步扩展能力范围。

### 5.2 职责边界

| 能力组件 | 主要职责 | 不承担 |
| --- | --- | --- |
| 统一入口（编排） | 理解问题、最小澄清、继承上下文、分发处理、整合结果、失败分流 | 不直接改写业务口径，不绕过权限，不承担具体取数细节 |
| 查询能力 | 负责清单、明细、指标、趋势、对比等结果查询 | 不做原因判断，不承担订单全过程追溯 |
| 诊断分析能力 | 基于已查结果解释波动、拆解影响方向并给出建议 | 不在缺少依据时直接下确定结论，不替代通用查数 |
| 追溯能力 | 围绕单个订单或制造订单返回过程、质量履历与链路摘要 | 不输出归因结论，不替代统一入口 |

### 5.3 调用关系

推荐形态：用户提问 -> 统一入口理解与最小澄清 -> 查询能力 / 诊断分析能力 / 追溯能力 -> 统一入口整合回显 -> 支持继续追问

V1 本期做法：

1. 对外仅保留一个统一入口。
2. 对内优先按首批问题清单和明确规则分发到相应能力。
3. 无论底层由哪类能力处理，最终都按统一的 5 项内容回显。

### 5.4 本期要求

1. V1 对外保持一个统一入口。
2. V1 对内按查询、诊断分析、追溯三类能力协同处理。
3. 本期允许采用“首批问题清单 + 规则分发”方式保障交付速度，不以完全自动编排作为上线前提。
4. S-04 及类似订单过程、质量履历、跨对象追溯问题，由追溯能力承接。
5. 诊断分析默认复用上游查询结果；确有必要时，允许做同口径的少量补充查询。
6. 在追溯场景中，用户泛称“订单”时，V1 默认按“生产订单”理解；用户显式说“制造订单”或对象编号可稳定识别时按实际对象处理。
7. V1 仅覆盖首批清单问题，清单外问题不纳入演示与验收。

| AC-08 | 首批问题清单通过率 | S-01~S-13 整体通过率达标 |
| AC-09 | 连续追问链路通过 | C-01 四步均可执行且上下文一致 |
| AC-10 | 工作台协同边界 | 可查同一计算说明 KPI；不出现首页卡片能力 |
| AC-11 | 分析证据约束 | 分析结论有可见证据支撑；不把相关性直接表述为确定因果 |
| AC-12 | 追溯链路完整性约束 | 定制追溯结果需显式展示已覆盖链路与缺失链路；不得编造全过程 |

### 6.2 基线测试规则

| 编号 | 测试要求 | 通过标准 |
| --- | --- | --- |
| RQ-01 | 首批问题清单全量测试（S-01~S-13） | 通过率 >= 90% |
| RQ-02 | 连续追问链路测试（C-01） | 4步连续通过且上下文一致 |
| RQ-03 | 每条结果展示计算说明、时间窗口、组织范围、数据来源、数据更新时间 | 页面可见上述信息 |
| RQ-04 | 图表问题符合展示要求 | 与第3章约定图表类型一致 |
| RQ-05 | 权限与异常分支必测 | 越权不泄露；异常有明确提示 |
| RQ-06 | 工作台协同边界必测 | 可查同一计算说明KPI；不产生首页卡片能力 |
| RQ-07 | 试点角色抽测 | 每个试点角色至少1条问题正确返回 |
| RQ-08 | 分析结果证据约束测试 | 归因/建议必须能回溯到上游结果或受限二次取数证据 |
| RQ-09 | 定制追溯链路缺口测试 | 对象不存在、命中多对象、链路缺失三类场景均有明确提示 |
| RQ-10 | 关键月度指标结果一致性检查 | 月度汇总结果与订单明细抽检一致 |

说明：本期验收关注问题处理结果、权限控制与规则正确性，不以是否实现完全自动分发能力作为通过条件。

### 6.3 推荐测试分层（V1）

V1 推荐采用“三层检查”的方式：

| 检查层 | 主要对象 | 建议方式 | 主要目的 |
| --- | --- | --- | --- |
| L1 业务场景验收 | 统一入口 | 业务用例验收 + 抽测 | 验证首批问题、连续追问、结果整合和边界分流是否符合预期 |
| L2 查询结果稳定性检查 | 查询能力 | 固定用例回归 + 结果抽检 | 防止字段、样例或数据接口变化后结果回归 |
| L3 诊断与追溯重点抽测 | 诊断分析能力、追溯能力 | 重点场景抽测 | 验证原因表述是否审慎、追溯链路是否完整、异常提示是否清晰 |

执行原则：

1. 对外验收以统一入口为主，不要求业务人员分别验收每个能力。
2. 查询结果要持续做固定用例回归，重点关注列表、指标、空结果和异常分支。
3. 月度汇总结果应能与订单明细相互印证。
4. 诊断与追溯重点关注边界、依据和异常提示，不以覆盖所有清单外问题为目标。

3. 沉淀常见分析场景模板，提升连续追问和诊断体验。
4. 补充更易复用的演示与验收材料，降低联调和培训成本。
5. 在权限、口径和结果可控前提下，再逐步扩大清单外问题覆盖范围。

## 附录A：V1设计原理与业界借鉴

### A.1 统一入口与子 Skill 协同原理

V1 推荐采用“统一入口 + 子 Skill 执行”的形态，核心原因如下：

1. 用户问题天然混合了查询、追问、诊断与追溯，不适合要求用户自己判断调用哪个 Skill。
2. 若查询 Skill、分析 Skill、追溯 Skill 平级直接抢触发，容易出现边界重叠、上下文继承混乱与验收口径不一致。
3. V1 首要目标是稳定交付首批清单问题，因此应优先保证“路由稳定、上下文一致、口径可回显、失败可分流”，而不是追求自治式多 Skill 编排。

推荐分层：

- 统一入口 / 编排 Skill：负责问题理解、最小澄清、上下文继承、子 Skill 路由、结果整合、失败分流
- 查询子 Skill：负责标准查询与指标查询
- 诊断子 Skill：负责“为什么、主因、贡献、影响、下一步动作”
- 追溯子 Skill：负责单订单 / 单制造订单的过程与质量履历追溯

### A.2 数据诊断原理

“数据分析”在本需求中不是泛化解释，而是**基于上游结果的诊断分析**，应遵循：

1. 先查后析：优先复用上游结果；只在验证分析结论必需时做受限补数。
2. 证据分级：只有汇总结果时只能确认现象；有维度拆解才能判断主要贡献项；有对象明细才能定位拖累对象；有事件/链路证据才能确认部分原因。
3. 对象分类：分析对象不只包含 KPI，还包括清单/对象结果、事件结果和追溯摘要。
4. 建议约束：建议必须由证据反推，且要绑定问题对象、预期收益、风险/代价与确认角色。

### A.3 V1 协调机制原则

V1 推荐采用：

- 规则映射优先，模型补充判断次之
- 标准输入输出契约优先，自治式多智能体次之
- 可解释和可追溯优先，覆盖率扩张次之

不建议 V1 直接以“一个简单 Agent 自动完美编排所有子 Skill”为上线前提。

### A.4 可借鉴的演进方向（用于后续迭代）

后续版本可重点关注以下方向：

1. **单入口优先**
   先用统一入口承接问题，再逐步扩展内部能力协同，避免一开始就把流程做得过于复杂。

2. **显式流程编排**
   当连续追问、失败分流和条件分支变复杂后，可把流程沉淀为更清晰的工作流。

3. **多角色协同**
   当任务需要跨多个分析阶段协同时，再考虑更细的角色协作方式。

4. **调用回看与评测**
   后续要逐步补齐调用回看、路由定位和回归评测能力。

## 附录B：当前4个 Skill 的最终定位表

下表可直接用于需求文档、方案说明或评审材料：

| 能力名称 | 定位 | 主要职责 | 典型问题 |
| --- | --- | --- | --- |
