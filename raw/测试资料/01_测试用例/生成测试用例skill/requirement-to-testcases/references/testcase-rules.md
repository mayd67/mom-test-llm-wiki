# 测试用例生成规则

使用这些规则把需求材料或功能测试点扩展成可直接落入 `测试用例模版.xlsx` 的测试用例。

## 一、推荐输入

可以接受以下任一输入，或混合输入：

- 需求文档
- 原型图说明
- 流程图说明
- 功能测试点清单

如果同时存在“需求材料 + 功能测试点”，优先以功能测试点为骨架，再用需求材料补全前置条件、步骤和预期结果。

## 二、模板字段

默认每条测试用例应映射到以下模板字段：

- 用例名称
- 所属模块
- 标签
- 前置条件
- 步骤描述
- 预期结果
- 编辑模式
- 备注
- 用例状态
- 责任人
- 用例等级

默认值建议：

- `编辑模式`：`TEST`
- `用例状态`：`未开始`
- `责任人`：留空，除非用户或材料明确指定
- `用例等级`：使用 `P0/P1/P2/P3`

## 三、字段映射建议

| 原始信息 | 模版字段 | 填写规则 |
| --- | --- | --- |
| 用例编号 + 用例标题 | 用例名称 | 优先写成 `TC-001 用例标题`；若无编号，可仅写标题 |
| 章节/功能点/模块层级 | 所属模块 | 使用路径形式，例如 `/产品/一级模块/二级模块/功能点` |
| 需求名称/文档名称/项目主题 | 标签 | 优先使用用户提供的需求标题或文档标题 |
| 前置条件 | 前置条件 | 单元格内按 `1.`、`2.` 编号，Markdown 中可用 `<br>` 分行 |
| 测试步骤 | 步骤描述 | 一个步骤一行；输出到模板时按行展开 |
| 预期结果 | 预期结果 | 与步骤逐行对应；不足的行可留空 |
| 来源测试点/场景类型/验证结果/待确认信息 | 备注 | 统一放入备注，便于追踪 |
| 优先级 | 用例等级 | 直接填写 `P0/P1/P2/P3` |

## 四、行展开规则

- 一个用例只验证一个核心目标。
- 一个步骤只写一个动作，避免一个步骤里混多个动作。
- 一个预期结果只写一个可验证结果，尽量与步骤或最终业务结果对应。
- 当一条用例有多步时，按模板拆成多行输出：
  - 首行：填写全部共享字段 + 第 1 步 + 第 1 个预期结果
  - 后续行：仅填写 `步骤描述`、`预期结果`
  - 其他列留空，保持与 Excel 合并单元格语义一致
- 当“步骤数”和“预期结果数”不一致时，按较大行数展开；缺失的一侧留空。

## 五、场景类型建议

| 场景类型 | 适用场景 |
| --- | --- |
| 功能 | 正常业务主流程 |
| 异常 | 非法输入、失败路径、错误处理 |
| 边界 | 上下限、临界值、空值、极端情况 |
| 权限 | 角色、菜单、按钮、数据权限 |
| 状态 | 状态流转、回退、非法跳转 |
| 数据 | 数据规则、计算、唯一性、映射关系 |

`场景类型` 默认不单独占列，写入 `备注` 即可；除非用户明确要求单独保留。

## 六、编写规则

- 前置条件只写执行前必须满足的条件。
- 预期结果要可验证，不要只写“成功”“正常”。
- 不要写成纯概念描述，例如：
  - 错误：测试提交功能
  - 正确：填写完整信息后点击提交按钮，验证状态更新为“已提交”
- 若同一功能在不同角色、不同状态、不同组织或不同配置下规则不同，应拆成不同用例。
- 若材料中已有测试点编号，应尽量保留到 `用例名称` 或 `备注` 中。

## 七、优先级建议

| 优先级 | 适用情况 |
| --- | --- |
| P0 | 主流程、阻塞型问题、核心业务不可用 |
| P1 | 重要流程、关键规则、较高业务影响 |
| P2 | 一般场景、普通异常、常规边界 |
| P3 | 低频场景、弱影响优化点 |

## 八、生成顺序建议

默认建议按以下顺序生成：

1. 主流程功能用例
2. 关键异常用例
3. 边界用例
4. 权限用例
5. 状态流转用例

## 九、待确认规则

如果以下信息在需求中缺失，但又影响用例完整性，应继续输出，并在 `备注` 中补 `待确认信息`：

- 精确状态值
- 角色权限细则
- 特定字段校验规则
- 错误提示文案
- 外部系统返回规则

## 十、输出要求

- 默认输出为与模板同列顺序的 Markdown 表格。
- 如工作区存在 `测试用例模版.xlsx` 且用户要求文件结果，应优先直接产出 `.xlsx` 文件。
- 不要再默认输出旧版“用例编号/用例标题/来源测试点/功能点/优先级/场景类型”表，除非用户明确要求兼容旧格式。
