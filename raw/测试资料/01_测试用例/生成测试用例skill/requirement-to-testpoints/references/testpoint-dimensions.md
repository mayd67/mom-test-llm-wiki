# 测试点拆分维度

使用以下维度把需求材料拆成可分析、可追踪的测试点。

## 一、核心拆分维度

| 维度 | 关注内容 | 常见示例 |
| --- | --- | --- |
| 功能流程 | 主用户路径和成功完成路径 | 新增、提交、审批、查询 |
| 异常流程 | 非法输入、失败路径、阻断场景 | 空值、重复数据、超时 |
| 边界条件 | 上下限、临界值、最小/最大输入 | 最大长度、最小值、最后一个状态 |
| 数据规则 | 格式、唯一性、依赖、计算逻辑 | 编码格式、必填组合、合计值 |
| 权限控制 | 角色、按钮、菜单、数据可见性 | 可见但禁用、隐藏操作 |
| 状态流转 | 合法与非法状态变化 | 草稿到已提交、驳回到草稿 |
| 交互反馈 | 消息、提示、确认、加载、空态 | 成功提示、报错提示 |
| 依赖与联动 | 上下游模块、共享对象、公共规则 | 配置影响列表、状态影响报表 |

## 二、按输入类型拆分

### 1. 输入是需求文档时

- 提取业务目标、前置条件、触发动作、业务规则、异常处理和结果。
- 对 `支持`、`应`、`可以`、`通常` 这类模糊表述保持警惕，必要时转成风险点或待确认项。

### 2. 输入是原型图时

- 重点看字段、默认值、占位提示、隐藏态、禁用态、按钮可用性、表格列和反馈区域。
- 留意未描述的校验规则、空状态和交互反馈。

### 3. 输入是流程图时

- 列出所有状态、触发动作、分支、循环、回退路径和终止状态。
- 识别图上看起来可达、但正文没有解释的跳转。

## 三、默认输出范围

默认应输出：

1. 测试点清单
2. 需求风险或缺口
3. 评审问题
4. 明显的跨模块影响提示
5. 待确认信息

## 四、拆分规则

- 一个测试点应描述一个清晰、可验证的行为。
- 一句话里如果包含多个动作或多个条件，应该拆开。
- 如果状态变化受角色、数据条件或来源渠道影响，按角色或条件继续拆分。
