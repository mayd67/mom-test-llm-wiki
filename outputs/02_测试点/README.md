# 测试点输出目录说明

`outputs/02_测试点/` 用于保存基于具体需求产出的测试点分析结果。

这里存放的是测试工作过程中的交付物，不是长期知识库页面。

## 目录结构

```text
02_测试点/
├── 01_待评审/
├── 02_已评审/
├── 03_已沉淀/
└── 测试点模板.md
```

## 目录用途

### 01_待评审

存放刚完成初稿、尚未经过评审确认的测试点文档。

适用场景：

- 刚基于需求、设计、原型完成测试点分析
- 待产品、研发、测试组内评审
- 仍可能补充边界、异常、权限和数据链路测试点

### 02_已评审

存放已经过评审确认、可继续进入测试用例编写阶段的测试点文档。

适用场景：

- 需求评审已结束
- 测试范围已确认
- 可以基于该版本继续输出测试用例

### 03_已沉淀

存放已经从当前需求测试点中提炼出通用规则、并完成沉淀标记的文档。

说明：

- 这里仍然属于 `outputs/`，表示它最初是测试交付物
- 真正适合长期复用的共性知识，应再提炼后沉淀到 `wiki/`

## 工作流

```text
wiki中的需求文档
-> 编写测试点初稿
-> 保存到 01_待评审/
-> 评审确认
-> 移动到 02_已评审/
-> 编写测试用例
-> 提炼通用测试规则
-> 标记并移动到 03_已沉淀/
-> 将通用知识整理进入 wiki/
```

## 命名建议

```text
需求编号-需求名称-测试点-V1.0.md
需求编号-需求名称-测试点-YYYYMMDD.md
```

示例：

```text
DNW31500-智能问数-测试点-V1.0.md
DNW30301-计划调度-测试点-20260428.md
```

## 编写原则

- 每份测试点文档只聚焦一个需求或一个清晰范围。
- 测试点先覆盖功能主流程，再补业务规则、异常、权限、数据与状态链路。
- 需求文档默认从 `wiki/` 中读取和引用。
- 设计资料、数据模型资料优先引用 `wiki/` 中的原文镜像、导航页或整理页。
- 只有 `wiki/` 尚未覆盖时，资料来源才回退到 `raw/`。
- 不把测试用例步骤直接写成测试点，测试点更关注“测什么”。
- 测试点确认后，再进入测试用例编写阶段。
