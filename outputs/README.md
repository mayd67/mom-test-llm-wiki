# 测试交付物目录说明

`outputs/` 用于保存测试工程师日常工作产生的交付物。

本目录不是原始资料区，也不是结构化知识库：

- `raw/`：知识来源，保存需求文档、原型图、接口资料、会议纪要、历史测试资料等原始输入。
- `wiki/`：加工后的知识库，保存经过整理、提炼、关联和维护的结构化知识。
- `outputs/`：测试工作的交付物，保存基于 `raw/` 和 `wiki/` 生成或维护的工作成果。

## 目录结构

```text
outputs/
├── 01_需求评审/
├── 02_测试点/
├── 03_测试用例/
├── 04_测试执行/
├── 05_用户操作手册/
└── 99_归档/
```

## 目录用途

### 01_需求评审

保存需求评审阶段的交付物，例如：

- 需求评审问题清单
- 需求疑问与澄清记录
- 需求风险分析
- 需求可测性分析
- 评审结论与遗留问题跟踪

### 02_测试点

保存从需求、业务规则、流程图、原型和历史缺陷中分析出的测试点，例如：

- 按需求编号生成的测试点
- 按功能模块整理的测试点
- 业务规则测试点
- 异常流程测试点
- 权限与数据范围测试点

推荐在该目录下继续区分：

- `01_待评审/`
- `02_已评审/`
- `03_已沉淀/`

### 03_测试用例

保存正式可执行或可导入测试管理平台的测试用例，例如：

- 功能测试用例
- 流程测试用例
- 接口测试用例
- 权限测试用例
- 异常测试用例
- 回归测试用例

### 04_测试执行

保存测试执行过程中的阶段性结果，例如：

- 测试计划
- 测试执行记录
- 缺陷统计
- 回归验证记录
- 测试报告

### 05_用户操作手册

保存面向用户、实施、交付或内部培训的操作说明，例如：

- 按模块编写的功能操作手册
- 按角色编写的用户操作手册
- 操作步骤截图说明
- 常见问题与异常处理说明
- 可发布版本手册

### 99_归档

保存已完成版本、已交付项目或历史阶段的交付物归档，例如：

- 按版本归档
- 按项目归档
- 按发布时间归档
- 已废弃但需要留痕的历史交付物

## 使用原则

- `outputs/` 中的文件应面向交付和执行，避免堆放未经整理的原始资料。
- 交付物引用的需求、原型、接口、会议纪要等来源应能追溯到 `raw/`。
- 交付物中沉淀出的稳定规则、模板、测试策略、缺陷模式，应反哺到 `wiki/`。
- 临时草稿可以放在对应目录下，但定稿文件应在文件名中标明需求、模块、版本或日期。
- 已完成或不再活跃的交付物应移动到 `99_归档/`，避免干扰当前工作。

## 推荐命名

```text
需求编号-模块名称-交付物类型-版本.md
需求编号-模块名称-交付物类型-YYYYMMDD.md
项目名称-版本号-交付物类型-YYYYMMDD.xlsx
```

示例：

```text
DNW31500-智能问数-测试点-V1.0.md
DNW31500-智能问数-测试用例-20260428.xlsx
MOM3.0-V1.2-回归测试报告-20260428.md
```
