# 测试工程师 LLM Wiki 知识库搭建

编制日期：2026-04-27   
项目名称：测试工程师 LLM Wiki 本地知识库建设

## 一、项目描述

### 1.1 项目背景

本项目围绕测试团队知识沉淀、业务资料复用和私有化 LLM 问答检索场景建设，目标是将分散的需求文档、产品资料、接口资料、测试规范和历史经验，统一沉淀为可检索、可追溯、可持续维护的本地知识库工程。

项目采用“原始资料层 + 结构化知识层 + 导航维护层”的建设思路：

- 原始资料层：统一接收需求文档、项目文档、产品资料、接口资料、测试资料与截图附件等输入材料。
- 结构化知识层：按照测试团队使用场景，将内容沉淀到通用规范、测试标准、业务系统、技术手册、问题沉淀、专项测试、新人赋能等主题目录。
- 导航维护层：通过统一首页、总览页、操作日志和元数据同步机制，确保知识库具备可导航、可维护、可持续扩展能力。

### 1.2 项目目标

项目当前目标主要包括以下四个方面：

1. 建立测试团队统一知识底座，支撑规范、模板、业务知识和经验沉淀。
2. 建立适配私有化 LLM 的结构化知识组织方式，提升问答检索准确率和追溯能力。
3. 建立原始资料入库、结构化拆解、导航更新、日志留痕的标准化维护流程。
4. 建立面向 MOM 业务系统的重点知识专题，服务测试分析、新人培训和问题自查。

### 1.3 建设范围

当前项目建设范围覆盖两大部分：

#### （1）资料治理范围

- `raw/需求文档/`
- `raw/项目文档/`
- `raw/产品资料/`
- `raw/接口资料/`
- `raw/测试资料/`
- `raw/测试流程规范/`
- `raw/截图附件/`

#### （2）知识库建设范围

- `wiki/01_通用规范`：组织职责、研发流程、制度与安全要求
- `wiki/02_测试标准&模板`：测试规范、缺陷管理、复用模板
- `wiki/03_业务系统`：以 MOM 为核心的系统总览、模块业务、接口与数据字典、原始资料镜像
- `wiki/04_技术手册`：环境、数据库、工具、代码管理
- `wiki/05_问题沉淀库`：环境问题、故障复盘、高频报错
- `wiki/06_专项测试`：功能、接口、性能、安全测试
- `wiki/07_新人赋能`：学习路径、工具配置、术语词典

### 1.4 实施思路与技术路线

本项目不是单纯堆放文档，而是采用“资料治理 + 自动化构建 + 结构化编排 + LLM 检索适配”的实施模式推进：

1. 先统一接收原始资料，保证输入材料不丢失、可追溯。
2. 再通过脚本将需求、产品资料、原始 Markdown、Office 文档、附件等转化为可进入 Wiki 的中间成果。
3. 对重点主题进行结构化拆解，形成总览页、导航页、地图页、数据字典页和专题页。
4. 通过 `wiki/index.md` 和 `wiki/log.md` 维持全局导航与变更留痕。
5. 通过 `scripts/sync_wiki_meta.py` 保障知识库元数据、首页导航和日志同步更新。

### 2.1 已实施计划内容

#### （1）基础架构与规则建设

- 完成知识库初始化，建立项目基础目录、规则文件和维护文件。
- 完成知识库 Schema 设计，明确页面类型、目录归属、补全标准、检索策略和更新策略。
- 将知识库结构调整为七大目录，适配企业内部测试团队长期建设与私有化 LLM 使用场景。

#### （2）原始资料治理建设

- 将原始资料统一归档到 `raw/` 目录，按测试工作输入进行分类管理。
- 将需求文档、产品资料、接口资料、项目材料、测试资料和附件实现统一入口管理。
- 建立“原始资料只存 raw、结构化知识沉淀到 wiki”的边界规则，降低资料混放风险。

#### （3）自动化建设能力

已建设 9 个核心脚本，形成自动化能力：

- `scripts/sync_wiki_meta.py`：自动刷新 `wiki/index.md`，同步写入 `wiki/log.md`，维护全局导航和变更记录。
- `scripts/build_requirements_wiki.py`：基于 `raw/需求文档/` 自动生成 MOM 需求拆解页、需求地图和导航页。
- `scripts/build_kmmom_product_docs_wiki.py`：基于 `raw/产品资料/km-mom-docs/` 生成产品资料结构化页面和阶段资料地图。
- `scripts/build_kmmom_data_dictionary.py`：基于数据模型资料生成数据库表、枚举和业务状态索引。
- `scripts/build_raw_markdown_mirror.py`：将原始 Markdown 资料镜像到 Wiki 中，形成可浏览的原始资料入口。
- `scripts/build_raw_office_mirror.py`：将 Word、Excel 等 Office 资料转换为可沉淀页面。
- `scripts/build_raw_attachment_catalog.py`：为附件和图片类资料生成索引。
- `scripts/build_wiki_coverage_matrix.py`：用于统计原始资料到结构化知识的覆盖关系。
- `scripts/restructure_wiki_for_team_template.py`：用于知识库目录重构和模板化调整。

#### （4）业务专题建设

- 已将 MOM 业务系统作为当前重点建设对象。
- 已围绕 MOM 建设系统总览、需求分析总览、产品资料总览、原始资料镜像、接口与数据字典等重点入口。
- 已形成从需求、产品资料到表结构/枚举、模块导航、业务 FAQ 的多层级知识组织方式。

### 2.2 下一阶段实施建议

为便于后续汇报和持续建设，建议下一阶段按以下计划推进：

1. **补齐覆盖率治理**：输出原始资料入库覆盖矩阵，明确“已结构化/待补充/仅镜像”状态。
2. **完善问题沉淀与专项测试**：提升 `05_问题沉淀库`、`06_专项测试`、`07_新人赋能` 的内容密度。
3. **强化业务闭环**：将需求拆解页、产品资料页、数据字典页和测试资产页建立更完整的双向链接。
4. **形成可复制模板**：沉淀一套可复制到其他业务系统的 LLM Wiki 建设模板和脚本流程。
5. **接入团队日常使用**：将知识库应用到测试分析、问题排查、新人培训和私有化问答检索场景中。
