# LLM WIKI操作使用指南-生成测试用例

本文档用于说明：测试工程师如何使用当前 LLM WIKI 知识库，完成“从需求到测试用例”的完整操作。

本文以“生成测试用例”为主线，兼顾以下几个问题：

- 资料应该从哪里找
- 先看需求还是先看设计
- 测试点和测试用例分别放哪里
- wiki、raw、outputs 三者分别怎么用

## 一、先理解这套知识库怎么分层

当前项目分成三层：

- `wiki/`：知识整理层，日常使用的主入口
- `raw/`：原始资料层，作为追溯依据，不作为默认读取入口
- `outputs/`：测试交付层，保存当前需求实际产出的测试点和测试用例

如果目标是生成测试用例，默认操作顺序是：

```text
先看 wiki
-> 需要追溯原文时再看 wiki 中的原文镜像
-> 当前需求交付物输出到 outputs
```

## 二、生成测试用例的标准路径

当前知识库已经形成的标准路径是：

```text
需求文档
-> 设计与数据模型资料
-> 生成测试点
-> 测试点评审
-> 生成测试用例
-> 测试用例评审
-> 导入 MeterSphere
```

## 三、第一步：找到需求文档

需求文档统一从这里进入：

`wiki/01_MOM产品相关文档/01_产品需求文档/`

当前一般分三类：

- `01_标准功能/`
- `02_MOM-AI/`
- `03_优化需求/`

使用方法：

1. 先确认需求编号，比如 `DNW30710`
2. 到对应目录中找到需求文档
3. 优先看以下内容：
   - 业务目标
   - 角色
   - 主流程
   - 范围与非范围
   - 验收标准

如果是录入式、配置式、流程式需求，建议额外留意：

- 查询条件
- 列表字段
- 明细字段
- 导入导出
- 删除与批量操作

## 四、第二步：查设计和数据模型

设计和数据模型默认从这里进入：

`wiki/01_MOM产品相关文档/02_数据模型/`

推荐入口：

- `00_数据模型总览.md`
- `09_原始文档导航.md`

### 1. 快速理解时

优先看：

- `00_数据模型总览.md`
- `01_核心数据模型地图.md`
- `02_模块设计地图.md`

### 2. 查正式设计依据时

优先看：

`02_原始文档/03-development/`

里面按原结构保留了：

- `datamodel-design/`
- `module-design/`
- `tech-design/`
- `webapi/`

如果你的目的是生成测试用例，通常最常用的是：

- `module-design/`：看主流程、规则、边界、兼容、回归影响
- `datamodel-design/`：看对象、字段、状态、回写、主从关系
- `webapi/`：看接口结构、错误码、返回结构

## 五、第三步：先生成测试点，不要直接写用例

测试点和测试用例不要混着做。

先生成测试点，放在：

`outputs/02_测试点/01_待评审/`

模板参考：

`outputs/02_测试点/测试点模板.md`

生成测试点前，建议同时参考：

- `wiki/00_方法规范/测试点生成与评审流程.md`
- `wiki/00_方法规范/测试点评审检查清单.md`

测试点至少应覆盖：

- 功能测试点
- 业务规则测试点
- 异常测试点
- 权限测试点
- 数据与状态测试点
- 接口与联动测试点
- 回归关注点

## 六、第四步：评审测试点

测试点不是写完就能直接生成测试用例，建议先评审。

评审时重点看五类依据：

1. 需求文档
2. 设计文档
3. 数据模型
4. 接口资料
5. 测试经验

评审检查清单见：

`wiki/01_MOM产品相关文档/00_方法规范/测试点评审检查清单.md`

评审通过后，把测试点移到：

`outputs/02_测试点/02_已评审/`

只有进入“已评审”状态，才建议继续生成测试用例。

## 七、第五步：基于已评审测试点生成测试用例

测试用例统一输出到：

`outputs/03_测试用例/`

建议命名：

- `需求编号-需求名称-测试用例-V1.0.md`

生成测试用例时，输入材料应包括：

- `outputs/02_测试点/02_已评审/` 中的测试点
- `wiki/01_产品需求文档/` 中的需求文档
- `wiki/02_数据模型/` 中的设计与数据模型资料
- `wiki/05_产品操作手册/` 中的操作路径

## 八、第六步：按 MeterSphere 口径整理测试用例

如果测试用例最终要导入 MeterSphere，请统一参考：

`wiki/01_MOM产品相关文档/04_测试用例/03_测试用例导入模板/测试用例模板说明.md`

当前统一口径如下：

- 标签：`需求编号-需求名称`
- 编辑模式：`TEXT`
- 用例状态：`未开始`
- 责任人：留空

字段统一为：

- 用例名称
- 所属模块
- 标签
- 前置条件
- 步骤描述
- 预期结果
- 编辑模式
- 备注
- 用例状态
- 责任人
- 用例等级

### 用例等级规则

- `P0`：阻断主流程、影响上线
- `P1`：重要规则、高频功能、关键联动
- `P2`：辅助功能、低频场景、一般边界

## 九、第七步：评审测试用例

测试用例评审重点不再是“覆盖什么”，而是“能不能执行”。

重点看：

- 用例是否覆盖已评审测试点
- 前置条件是否清晰
- 步骤是否可执行
- 预期是否可验证
- 模块路径、标签、状态、等级是否符合导入模板规则
- 是否存在重复用例或空泛预期

相关方法见：

`wiki/01_MOM产品相关文档/00_方法规范/测试用例生成与评审流程.md`

## 十、以一个真实需求为例

以 `DNW30710-物流管理` 为例，当前这套知识库的使用方式是：

1. 到 `wiki/01_产品需求文档/01_标准功能/` 读取 `DNW30710-物流管理.md`
2. 到 `wiki/02_数据模型/09_原始文档导航.md` 找 `03-development/module-design/DNW30710-物流管理设计文档.md`
3. 到 `outputs/02_测试点/` 生成测试点初稿
4. 按评审意见修订测试点，形成已评审版本
5. 到 `outputs/03_测试用例/` 生成测试用例初稿
6. 按 MeterSphere 模板口径整理字段和等级

这就是当前知识库最典型、也最推荐的使用路径。

## 十一、常见误区

### 误区1：直接从 raw 找资料

不推荐。

正确做法：

- 先从 `wiki` 读
- 需要追溯原文时，再读 `wiki` 中的原文镜像

### 误区2：跳过测试点，直接写测试用例

不推荐。

正确做法：

- 先测试点
- 再测试用例

### 误区3：测试用例不按导入模板口径输出

不推荐。

正确做法：

- 统一按 MeterSphere 模板字段和默认值整理

### 误区4：把所有临时交付物都沉淀进 wiki

不推荐。

正确做法：

- 当前需求交付物放 `outputs`
- 共性方法和规则才沉淀到 `wiki`

## 十二、推荐的日常使用顺序

如果你每天都在用这套知识库，建议固定顺序：

1. 打开 `wiki/index.md`
2. 进入 `01_产品需求文档`
3. 进入 `02_数据模型`
4. 根据需要进入 `00_方法规范`
5. 输出测试点到 `outputs/02_测试点`
6. 输出测试用例到 `outputs/03_测试用例`

## 十三、总结

当前这套 LLM WIKI 知识库，已经能够支持测试工程师完成以下动作：

- 找需求
- 查设计
- 查数据模型
- 生成测试点
- 评审测试点
- 生成测试用例
- 按 MeterSphere 口径整理测试用例

最关键的使用原则只有一句话：

**需求、设计、数据模型从 wiki 取；当前交付物往 outputs 放；共性方法再沉淀回 wiki。**
