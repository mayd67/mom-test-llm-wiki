# 测试用例生成与评审流程

本文档只聚焦测试用例阶段，说明如何基于已评审测试点生成测试用例、评审测试用例，并按 MeterSphere 导入模板统一输出。

## 一、原则

- 测试用例必须基于 `outputs/02_测试点/02_已评审/` 中的测试点生成。
- 需求文档、设计文档、数据模型仍默认从 `wiki/` 中读取。
- 测试用例输出必须遵循 MeterSphere 模板口径。
- 生成测试用例时，既要覆盖测试点，也要可执行、可导入、可追溯。

## 二、整体工作流

```text
已评审测试点
-> 读取 wiki 中的需求/设计/数据模型资料
-> 生成测试用例初稿
-> 输出到 outputs/03_测试用例/
-> 测试用例评审
-> 修订版本
-> 形成可执行/可导入版本
-> 提炼共性规则
-> 沉淀到 wiki/
```

## 三、测试用例生成流程

### 1. 前置条件

只有测试点进入“已评审”状态后，才建议继续编写测试用例。

原因：

- 测试点解决“测什么”
- 测试用例解决“怎么测”

如果前一层没确认，后一层很容易返工。

### 2. 输入材料

编写测试用例时，应同时参考：

- `outputs/02_测试点/02_已评审/` 中的测试点
- `wiki/01_产品需求文档/` 中的需求文档
- `wiki/02_数据模型/` 中的设计原文镜像、导航页和专题页
- `wiki/05_产品操作手册/` 中的操作路径说明
- 历史测试用例样例
- 测试数据准备说明

### 3. 输出位置

测试用例初稿统一输出到：

`outputs/03_测试用例/`

推荐命名：

- `需求编号-需求名称-测试用例-V1.0.md`
- `需求编号-需求名称-测试用例-YYYYMMDD.md`

如需导入 MeterSphere，统一参考：

- `wiki/01_MOM产品相关文档/04_测试用例/03_测试用例导入模板/测试用例模板说明.md`

### 4. 测试用例拆分建议

测试用例通常按以下方式拆分：

- 正常流程
- 异常流程
- 权限场景
- 数据与状态场景
- 接口与联动场景
- 回归场景

### 5. MeterSphere 输出口径

测试用例输出字段统一为：

- 用例名称
- 所属模块
- 标签
- 前置条件
- 步骤描述
- 预期结果
- 编辑模式
- 备注
- 用例状态
- 责任人
- 用例等级

固定默认值：

- 标签：`需求编号-需求名称`
- 编辑模式：`TEXT`
- 用例状态：`未开始`
- 责任人：留空

### 6. 用例等级规则

- `P0`：主流程核心功能、阻断业务闭环的能力
- `P1`：重要规则、高频功能、关键联动和常见异常
- `P2`：辅助功能、低频场景、一般边界和提示类验证

## 四、测试用例评审流程

### 1. 评审目标

确认测试用例是否已经把测试点落成可执行步骤，并且：

- 预期清晰
- 数据可准备
- 结果可验证
- 格式可导入

### 2. 评审重点

- 用例是否覆盖已评审测试点
- 用例步骤是否可执行
- 预期结果是否可验证
- 前置条件和测试数据是否清晰
- 一个用例的粒度是否合适
- 是否存在重复用例、空泛预期或不可操作步骤
- 是否区分正常、异常、权限、联动等场景
- 是否符合 MeterSphere 模板字段和默认值规则

### 3. 评审后处理

- 评审通过：继续进入执行或导入阶段
- 存在问题：修订版本并重新评审
- 有价值的共性规则：沉淀回 `wiki/00_方法规范/` 或 `wiki/04_测试用例/`

## 五、适合进入 wiki 的内容

适合沉淀到 `wiki/` 的通常是：

- 通用测试用例模板
- 用例等级规则
- MeterSphere 导入口径
- 模块级高频测试用例样式
- 常见可执行步骤写法

不建议原样搬进 `wiki/` 的通常是：

- 单次需求的临时测试用例草稿
- 未评审版本
- 只适用于某次版本的临时导入文件
