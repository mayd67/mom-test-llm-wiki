# 测试点生成与评审流程

本文档只聚焦测试点阶段，说明如何从 `wiki` 资料生成测试点、评审测试点，并形成可继续支撑测试用例编写的已评审版本。

## 一、原则

- 需求文档默认从 `wiki/01_产品需求文档/` 读取。
- 设计文档和数据模型默认从 `wiki/02_数据模型/` 读取。
- `outputs/02_测试点/` 存当前需求的测试点交付物。
- `wiki/` 存方法、规则、模板和沉淀知识，不直接替代测试交付物。

## 二、整体工作流

```text
wiki中的需求文档 / wiki中的设计资料 / wiki中的数据模型资料
-> 测试理解
-> 生成测试点初稿
-> 输出到 outputs/02_测试点/01_待评审/
-> 测试点评审
-> 修订版本
-> 移动到 outputs/02_测试点/02_已评审/
-> 提炼共性规则
-> 沉淀到 wiki/
```

## 三、测试点生成流程

### 1. 输入材料

生成测试点前，至少应读取以下资料：

- `wiki/` 中的需求文档
- `wiki/` 中的原型图、界面方案或业务流程整理
- `wiki/` 中的模块设计文档或原文镜像
- `wiki/` 中的数据模型、状态、枚举、初始化规则
- 历史缺陷或历史测试资料（若存在）

默认规则：

- 需求文档优先从 `wiki/01_产品需求文档/` 读取
- 设计文档和数据模型优先从 `wiki/02_数据模型/` 读取
- 只有当 `wiki` 尚未覆盖对应资料时，才回到 `raw/`

### 2. 输出位置

测试点初稿统一输出到：

`outputs/02_测试点/01_待评审/`

推荐命名：

- `需求编号-需求名称-测试点-V1.0.md`
- `需求编号-需求名称-测试点-YYYYMMDD.md`

### 3. 测试点结构

一份完整的测试点文档，至少应包含：

- 基本信息
- 资料来源
- 需求理解
- 功能测试点
- 业务规则测试点
- 异常测试点
- 权限测试点
- 数据与状态测试点
- 接口与联动测试点
- 回归关注点
- 测试数据准备说明
- 风险与待确认项

### 4. 生成测试点时的思路

先覆盖主流程，再补齐下面几类内容：

- 页面和操作主流程
- 字段和参数规则
- 状态流转
- 主单/明细关系
- 上下游单据和对象联动
- 默认值、兜底值、空值兼容
- 权限和组织隔离
- 历史数据兼容
- 回归影响范围

## 四、测试点评审流程

### 1. 评审目标

测试点评审不是检查格式，而是确认：

- 测试点是否覆盖完整
- 测试点是否符合需求和设计口径
- 测试点是否能支撑后续测试用例编写

### 2. 评审依据

测试点评审依据统一参考：

- `wiki/01_产品需求文档/`
- `wiki/02_数据模型/02_原始文档/03-development/`
- `wiki/02_数据模型/01_核心数据模型地图.md`
- `wiki/00_方法规范/测试点评审检查清单.md`

### 3. 常见评审发现类型

- 主数据维护能力漏测
- 状态流转口径和需求不一致
- 查询视角、筛选条件、列表展示规则漏测
- 导入失败、批量处理失败、原子性规则漏测
- 只测创建成功，不测失败语义和边界条件

### 4. 评审后处理

- 若需要修改：继续留在 `01_待评审/`
- 若评审通过：移动到 `outputs/02_测试点/02_已评审/`
- 若其中有通用规则可复用：后续提炼后沉淀到 `wiki/`

## 五、适合进入 wiki 的内容

适合沉淀到 `wiki/` 的通常是：

- 通用测试点生成方法
- 模块级测试关注点
- 高频业务规则
- 状态流转规则
- 常见异常场景
- 测试数据准备路径

不建议原样搬进 `wiki/` 的通常是：

- 某次需求的草稿测试点
- 某次迭代的临时修订版
- 未评审完成的测试输出物
