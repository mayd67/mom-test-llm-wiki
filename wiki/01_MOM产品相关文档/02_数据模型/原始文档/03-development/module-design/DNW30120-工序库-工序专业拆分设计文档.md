# MES 多专业拆分设计说明

## 1. 文档目标

本文档用于收敛 MES 多专业拆分的最终设计结论。

当前系统的专业判断仍然主要建立在组织/工厂类型上，本次调整只做一件核心事情：

- 把“专业真值”从组织挪到工序

同时明确两条边界：

- 不改组织归属、权限、数据范围等既有模型
- 本阶段不把专业快照到任务上

本文重点回答三个问题：

- 现有按组织拆分会带来什么影响
- 专业挂到工序后，数量和产出比应该怎么算
- 订单级操作需要怎样改造

## 2. 当前机制与核心矛盾

### 2.1 当前机制

当前多数核心流程仍然沿用如下模式：

`bizOrg.factoryType -> @FactoryType 实现类`

这意味着系统默认认为：

- 一张制造订单在同一时刻只属于一个专业
- 工艺展开、工艺变更、批次/序列号、指定工艺路线、订单分卡等订单级操作，会先选定一个专业实现，再整单执行
- 数量口径也默认跟着这个“整单单专业”假设走

### 2.2 当前机制的问题

当一条工艺路线同时包含机加、装配等不同执行语义时，上述假设会直接产生冲突：

- 业务语义冲突：同一订单既想按机加算产出比，又想按装配不算产出比，整单无法只选一种规则
- 数量口径冲突：如果一个工序是机加、下一个工序是装配，继续按整单专业算，展开数量、报工数量、在制数量会前后不一致
- 入口职责冲突：订单级入口本不该再承担“这张单到底是机加还是装配”的判断，而应该先跑公共流程，再把差异下沉到工序级

## 3. 最终设计结论

### 3.1 专业真值挂在工序

最终以 `RoutingProcessLink.specialtyCode` 作为专业真值。

规则如下：

- 一道工序只能有一个专业
- 一张订单可以跨多个专业
- 如果现场业务上看起来是“同一道工序既有机加又有装配”，在数据模型上必须拆成两道工序，否则数量和执行语义都无法唯一确定

### 3.2 组织继续保留，但不再作为专业分发依据

`bizOrg` 仍然保留以下职责：

- 归属
- 权限
- 数据范围
- 默认配置

现有 `bizOrg.factoryType` 建议短期保留，但语义降级为：

- 历史兼容字段
- 默认值补齐字段
- 迁移期间兜底字段

它不再作为以下能力的真实判断依据：

- 工艺展开
- 工艺变更
- 批次/序列号
- 指定工艺路线
- 订单分卡
- 报工、在制、转序等工序执行分发

### 3.3 本阶段不做任务专业快照

本阶段不建议把专业快照到 `ManuTask` 上，任务仍通过其关联工序解析专业。

原因：

- 本次目标只是把专业从组织挪到工序，尽量不扩大改造面
- 任务快照一旦引入，就要同时处理历史回填、变更同步、快照失效等额外问题
- 现有任务执行入口大多都能追溯到工序，只要补一个批量专业解析器即可

对应约束：

- 任务侧需要通过 `routingProcess -> specialtyCode` 批量解析专业，避免 N+1 查询
- 不能再直接以 `bizOrg.factoryType` 推断任务专业

### 3.4 工序专业必须具备版本稳定性

因为本阶段不做任务快照，所以 `RoutingProcessLink.specialtyCode` 必须被视为工艺版本属性，而不是可随意修改的运行期字段。

因此建议：

- 已发布、已展开、已生成任务的工艺版本，不允许直接原地修改工序专业
- 如果专业发生变化，应走工艺变更或新版本工艺路线
- 历史任务继续跟随历史工艺版本解释，不能因为主数据改动导致历史业务口径漂移

### 3.5 目标架构

建议形成三层结构：

- 订单级公共流程：负责统一入口、参数校验、对象装载、事务边界、状态流转
- 工序级专业策略：负责机加、装配等差异规则
- 公共数量引擎：负责展开、报工、分卡、WIP 中的输入/输出数量换算

一句话概括就是：

**整单走公共流程，差异下沉到工序专业，数量统一由公共引擎计算。**

## 4. 数量与产出比设计

### 4.1 基本原则

工艺展开时，不再先决定“整单走机加专业还是装配专业”，而是每道工序按自己的专业参与计算。

为避免混合工艺路线下数量口径混乱，引入统一概念：

`effectiveRate = 当前工序在数量换算中的有效产出比`

### 4.2 各专业的有效产出比

| 专业 | effectiveRate 规则 | 说明 |
| --- | --- | --- |
| `MACHINING` | `productivityRate` | 继续沿用机加现有产出比逻辑 |
| `ASSEMBLY` | `1` | 装配不按产出比放大或折减数量 |
| 其他专业 | 首期按 `1` 处理 | 后续需要时再扩展专属规则 |

同时增加一条主数据硬约束：

- 当工序专业为 `ASSEMBLY` 时，不允许填写产出比

也就是说：

- 装配工序的 `effectiveRate` 不是录入值，而是系统固定值 `1`
- 前端应禁用产出比录入
- 后端应做强校验，杜绝通过导入、接口、脚本绕过

这样可以从源头上避免“装配工序误填产出比，导致数量口径混乱”的问题。

### 4.3 展开数量公式

设订单计划完工数量为 `Q`，工艺路线共有 `n` 道工序，每道工序的有效产出比为 `r1...rn`。

则：

- 路线累计产出比 `routeProduct = r1 * r2 * ... * rn`
- 首道工序投入数量 `baseInputQty = Q / routeProduct`
- 第 1 道工序产出数量 `output1 = baseInputQty * r1`
- 第 `i` 道工序投入数量 `inputi = output(i-1)`
- 第 `i` 道工序产出数量 `outputi = inputi * ri`

这样有两个直接好处：

- 机加工序会按产出比放大前序投入量
- 装配工序因为 `ri = 1`，天然满足“投入数量 = 产出数量”

如果涉及舍入，统一沿用现有数量精度和舍入策略，由公共数量引擎集中处理，避免不同专业各算各的。

### 4.4 为什么这样算不会乱

如果一条路线是：

- 工序10：机加，产出比 `0.8`
- 工序20：装配，产出比按 `1`
- 工序30：机加，产出比 `0.9`

计划完工数量为 `100`，则：

- `routeProduct = 0.8 * 1 * 0.9 = 0.72`
- `baseInputQty = 100 / 0.72 = 138.888...`
- 工序10产出 `111.111...`
- 工序20投入和产出都为 `111.111...`
- 工序30产出 `100`

也就是说，装配工序本身不会扰乱机加数量链路，它只是一个 `effectiveRate = 1` 的工序节点。

### 4.5 报工、WIP、分卡数量统一规则

后续报工、在制流转、分卡都不应再自己发明一套“机加算法”或“装配算法”，而应统一基于当前工序的输入/输出关系计算。

统一建议：

- 当前工序如果是机加，则按该工序 `effectiveRate` 做输入和产出换算
- 当前工序如果是装配，则输入数量与产出数量相同
- 混合专业路线下，任何数量换算只围绕“当前工序”展开，不再使用“整单专业”做判断

这也是后续需要把 WIP 的数量转换能力从专业模块下沉到公共层的根本原因。

## 5. 订单级操作影响与改造建议

### 5.1 工艺展开

现状问题：

- 当前更偏向先根据组织或专业选实现，再整单展开

改造建议：

- 订单级入口仍保留一个公共展开流程
- 展开时先加载工艺路线全部工序，按工序专业计算整条路线数量
- 任务生成不再做“整单选机加还是装配”，而是每道工序带着自己的专业进入后续执行
- 专业模块只处理本工序需要补充的差异逻辑，比如装配关系、工序专属校验等

同时补充一条展开规则：

- 是否生成备料计划，不再按专业判断

统一建议改为：

- 只要该订单或工艺路线满足备料计划生成条件，展开时就生成备料计划
- 备料计划属于订单级公共能力，不再视为机加专属能力

这样可以避免“同一订单跨专业后，因为包含装配工序，导致整单不生成备料计划”的问题。

### 5.2 工艺变更

现状问题：

- 如果仍按整单专业处理，遇到只改某一道工序的专业会很别扭

改造建议：

- 工艺变更按“受影响工序”处理，而不是按“整单专业切换”处理
- 若工序专业发生变化，应视为工艺版本差异，触发受影响工序及任务的重算或重建逻辑
- 禁止对已发布版本直接改工序专业，必须通过工艺变更或新版本落地

### 5.3 批次或序列号

现状问题：

- 入口虽然是订单级，但实际绑定和校验对象是工序或工序任务

改造建议：

- 订单级入口可以继续保留
- 真正的校验与生成逻辑按目标工序专业处理
- 装配更关注 SN 与装配关系，机加更关注批量数量或过程批次，两者在入口层不必分两套流程，但在处理层必须按工序专业分流

### 5.4 指定工艺路线

现状问题：

- 过去容易把“指定路线”隐含为“指定专业”

改造建议：

- 指定工艺路线时不再推断整单专业
- 校验重点改为：路线工序是否都具备合法专业，是否存在同一工序混挂多个专业的脏数据
- 如果路线跨多个专业，这是合法场景，不应禁止

### 5.5 订单分卡

现状问题：

- 分卡入口是订单级，但后续拆分、报工拆分校验、拆分处理实际都落在工序任务层

改造建议：

- `getManuOrderSplitInfo` 继续走公共实现，不按专业分
- `manuOrderSplit` 按受影响工序的专业分组处理
- `validateManuTaskReportSplit` 按工序专业做校验
- `processManuTaskReportSplit` 按工序专业执行
- 分卡后的重新展开、报工数量重算，也统一走公共数量引擎，避免机加和装配各自重复实现一份数量换算

### 5.6 其他同类订单级触发点

除了用户直接关注的几个入口，还要同步注意这些“看起来不是专业功能、实际会触发专业逻辑”的场景：

- 订单下达后自动展开
- 工单流程结束后自动展开
- 分卡后自动重新展开
- 批量报工或批量在制处理

这些入口虽然是后台触发，但本质上仍然要遵循同样的工序专业规则，不能继续偷用组织专业。

## 6. 推荐改造方式

### 6.1 数据模型

优先新增或启用：

- `RoutingProcessLink.specialtyCode`

可选扩展：

- `RoutingProcessLink.executionMode`

明确不做：

- 本阶段不新增 `ManuTask.specialtyCode` 快照字段作为主方案

建议同步增加主数据校验规则：

- `ASSEMBLY` 工序禁止维护产出比
- `MACHINING` 工序如需数量换算，继续维护产出比
- 其他专业首期默认按不填产出比处理，由系统按 `effectiveRate = 1` 解释

### 6.2 核心解析与分发

建议新增统一能力：

- 工序专业解析器：从 `routingProcess` 批量解析专业
- 专业实现分发器：支持按 `specialtyCode` 获取实现，不再只接受 `bizOrg.factoryType`
- 工序分组器：把任务、报工、分卡等请求按工序专业分组执行

### 6.3 公共数量引擎

建议把以下逻辑收口到公共层：

- 工艺展开数量计算
- 报工数量换算
- WIP 数量转换
- 分卡数量拆分与回算

这样做之后：

- 机加只是 `effectiveRate = productivityRate`
- 装配只是 `effectiveRate = 1`
- 后续新增热处理、表处等专业，只需要补自己的 `effectiveRate` 或扩展规则，不需要重写一整套数量体系

### 6.4 兼容策略

短期兼容建议：

- `RoutingProcessLink.specialtyCode` 为空时，可在迁移期临时用 `bizOrg.factoryType` 或工序默认工作中心配置补齐
- 新建和变更工艺路线时，应强制要求工序具备专业
- 当所有核心入口完成改造后，再评估是否彻底弱化或移除组织专业字段的业务含义

## 7. 最终结论

### 7.1 设计结论

- 专业要从组织挪到工序，真实判断依据改为 `RoutingProcessLink.specialtyCode`
- 组织继续承担归属、权限、数据范围职责，但不再作为专业分发真值
- 本阶段不做任务专业快照，任务通过关联工序解析专业
- 工序专业必须具备版本稳定性，不能在已生效工艺上原地修改
- 所有订单级入口都应改成“公共流程 + 工序级专业分流 + 公共数量引擎”

### 7.2 一句话总结

**不是整张订单先选机加还是装配，而是每道工序按自己的专业执行；数量统一算，差异按工序处理。**
