# KMMOM 用户与组织架构设计方案

## 1. 概述

### 业务背景

在制造执行系统（MOM）中，对"用户"和"组织"的建模是系统设计的基础。由于制造企业存在大量临时工、外协人员等非正式员工，同时企业的管理架构和生产架构往往存在差异，因此需要建立清晰的概念模型和使用规范。

本设计方案旨在为 KMMOM 产品建立统一的**用户-组织**体系，解决以下核心问题：
- 如何处理临时工、外协人员等"无需登录系统"的业务参与者
- 如何同时支持"行政管理组织架构"和"生产业务组织架构"
- 如何在实体模型中正确使用用户和组织字段
- 如何基于业务组织实现多工厂数据权限隔离

### 设计目标

1. **概念清晰**：明确定义用户、行政组织、业务组织、工作中心的含义和边界
2. **模型简化**：用户既是系统账号也是人员信息载体，避免概念分裂
3. **灵活适配**：支持临时工、外协、第三方等多种用工形式
4. **架构稳定**：作为底层设计原则，确保数据模型的长期稳定性

### 设计原则

1. **用户统一管理**：User 既是系统账号也是业务人员，通过角色和功能权限控制访问范围
2. **双组织体系独立**：行政组织对接企业 HR，业务组织支撑制造流程，互不干扰
3. **数据权限基于业务组织**：多工厂、多车间数据隔离通过业务组织实现
4. **工作中心归属业务组织**：工作中心是生产资源，归属于业务组织管理

## 2. 核心概念定义

### 2.1 用户（User）

**定义**：系统账号与人员信息的统一载体，既用于登录认证，也用于记录业务执行者。

**核心特征**：
- **账号属性**：用户名、密码、用户类型、状态、锁定标记等，用于登录和权限控制
- **人员属性**：姓名、性别、出生日期、联系方式、邮箱、在职状态、在岗状态、证件号码等，用于业务记录
- **安全属性**：用户密级、密码验证失败次数、密码更新时间等，用于安全管理
- **权限控制**：通过角色和功能权限灵活控制用户的访问范围和操作能力

**用户类型（userType）**：

| 类型枚举 | 显示名称 | 说明 |
|---------|---------|------|
| `USER` | 普通用户 | 业务人员账号，参与制造业务流程 |
| `TRIAD` | 三员管理 | 系统管理员、安全管理员、审计管理员 |
| `SUPER` | 超级管理员 | 超级管理员账号，拥有最高权限 |

**用户状态（state）**：引用枚举 UserState
- INACTIVE（未激活）：用户账号已创建但未激活
- ACTIVE（正常）：用户账号正常可用
- DISABLED（已禁用）：用户账号被管理员禁用
- DELETED（已注销）：用户账号已注销

**在职状态（employedStatus）**：引用枚举 EmployedStatus
- EMPLOYED（在职）：员工在职
- TERMINATED（离职）：员工已离职
- REHIRED（返聘）：离职后返聘

**典型对象**：
- 正式员工：正常账号，通过角色授予业务功能权限
- 临时工：正常账号，通过角色控制限定访问范围（如仅能报工、查看任务）
- 外协人员：正常账号，通过角色控制限定访问范围
- 第三方检验员：正常账号，通过角色控制限定访问范围（如仅能录入检验结果）
- 系统管理员：管理账号（TRIAD/SUPER），仅进行系统管理操作，不参与制造业务

**实体模型**：`User`
- 表名：`SYS_USER`
- 主要字段：
  - 账号属性：`code`（用户账号）、`password`、`userType`、`state`（引用枚举 UserState）、`lockFlag`、`lockReason`
  - 人员属性：`name`（姓名）、`sex`（引用枚举 Sex）、`birthDate`、`phoneNumber`、`email`、`employedStatus`（引用枚举 EmployedStatus）、`dutyStatus`（引用枚举 DutyStatus）、`idNo`
  - 安全属性：`personLevel`（用户密级，引用枚举 PersonLevel）、`pwdErrNum`、`pwdUpdateTime`、`pwdResetFlag`、`pwdUseNum`

---

### 2.2 行政组织（Org）

**定义**：企业的管理架构，反映公司、部门等行政管理层级关系。

**核心特征**：
- 管理属性：用于企业行政管理、汇报关系、成本核算
- 数据来源：从企业 HR 系统或 ERP 系统集成
- 稳定性高：变更频率低，跟随企业组织调整
- 业务无关：不直接参与 MOM 系统的制造业务逻辑

**典型结构**：
```
集团公司
├── 华东分公司
│   ├── 研发部
│   ├── 生产部
│   ├── 质量部
│   └── 仓储部
└── 华南分公司
    ├── 生产部
    └── 质量部
```

**组织类型枚举**：

| 类型 | 说明 |
|------|------|
| 公司 | 法人实体、分公司，是企业法律主体、成本中心 |
| 部门 | 职能部门，是行政管理单位、汇报层级 |

**主要用途**：
- 用户的行政隶属关系（从 HR 系统同步）
- 财务成本核算的组织维度
- 审计和合规的组织视图
- 领导审批流的组织层级

**实体模型**：`Org`
- 表名：`SYS_ORG`
- 主要字段：`code`、`name`、`orgType`（组织类型：公司/部门）、`parentId`、`fullPath`、`enableFlag`

---

### 2.3 业务组织（BizOrg）

**定义**：制造执行的业务架构，反映工厂、车间、工段、班组等生产组织层级。

**核心特征**：
- 业务属性：直接参与 MOM 系统的制造业务逻辑
- 数据来源：在 MOM 系统中自主维护
- 变更灵活：根据产线调整、项目需要动态变化
- 权限载体：多工厂数据权限通过业务组织授予

**典型结构**：
```
集团制造中心
├── 工厂A（总装）
│   ├── 总装车间
│   │   ├── 工段1（总装一段）
│   │   │   ├── 白班班组
│   │   │   └── 夜班班组
│   │   └── 工段2（总装二段）
│   ├── 调试车间
│   └── 包装车间
├── 工厂B（机加）
│   ├── 机加车间1
│   │   ├── 数控工段
│   │   └── 普通机加工段
│   └── 机加车间2
└── 虚拟工厂（项目制）
    └── X项目协同组
```

**组织类型枚举**：

| 类型 | 说明 | 制造业务语义 | 典型层级 |
|------|------|------------|---------|
| 工厂（Factory） | 独立的生产基地 | 多工厂数据隔离的顶层单位、成本核算单位 | 1级 |
| 车间（Workshop） | 专业化生产单元 | 工艺专业划分（机加/装配/涂装）、产能统计单位 | 2级 |
| 工段（Section） | 车间内的作业单元 | 生产任务分配单位、班组管理层级 | 3级 |
| 班组（Team） | 最小作业组织 | 人员派工单位、绩效考核单位 | 4级 |

**说明**：
- 以上类型是典型的制造组织层级，系统支持灵活配置（如省略工段层级）
- 业务组织的层级划分基于实际制造管理需要，不强制四级
- 虚拟组织（如项目制协同组）可灵活定义类型

**主要用途**：
- 多工厂、多车间的数据隔离
- 生产任务的组织分配
- 资源（设备、工装、工作中心）的组织归属
- 产能、工时等生产数据的统计维度
- **数据权限的授予范围**（用户可以看哪些工厂的数据）

**实体模型**：`BizOrg`
- 表名：`SYS_BIZ_ORG`
- 主要字段：`code`、`name`、`bizOrgType`（组织类型：工厂/车间/工段/班组）、`parentId`、`fullPath`、`orgId`（关联行政组织）、`enableFlag`

---

### 2.4 两类组织的关系

#### 关联关系：N:1（业务组织 → 行政组织）

```
BizOrg (业务组织)  ────N:1────  Org (行政组织)
     │                            │
     │                            │
制造业务架构                   企业管理架构
```

**关联规则**：
- 一个业务组织通常对应一个行政组织（N:1）
- 业务组织通过 `orgId` 字段关联行政组织
- 关联是可选的，某些虚拟工厂可能不对应具体行政部门

**关联示例**：
```
业务组织                    行政组织
工厂A-机加车间1  ──────►   生产部
工厂A-机加车间2  ──────►   生产部
工厂A-质检车间   ──────►   质量部
虚拟工厂-X项目   ──────►   (无关联或关联项目部)
```

#### 职责边界

| 维度 | 行政组织（Org） | 业务组织（BizOrg） |
|------|----------------|-------------------|
| **数据来源** | 从 HR/ERP 集成 | MOM 系统维护 |
| **组织类型** | 公司、部门（行政管理层级） | 工厂、车间、工段、班组（制造业务层级） |
| **业务语义** | 管理架构、职能划分 | 制造架构、工艺划分 |
| **用户归属** | 用户的行政隶属（通过 OrgUser） | 用户的工作分配（通过 BizOrgUser） |
| **权限控制** | 不参与业务数据权限 | **数据权限主体**（多工厂隔离） |
| **成本核算** | 财务成本归集 | 制造成本归集 |
| **变更频率** | 低（跟随企业组织调整） | 相对较高（跟随产线调整） |
| **MOM业务** | **不参与**制造业务逻辑 | **直接参与**制造业务逻辑 |

#### 为什么需要双组织体系

**1. 组织结构差异**
- 行政上：按职能划分（研发部、生产部、质量部）
- 生产上：按工艺划分（机加工厂、装配工厂、涂装车间）
- 不强制对应：一个生产车间可能横跨多个行政部门

**2. 变更频率不同**
- 行政组织：稳定，企业组织架构调整才变化（低频）
- 业务组织：灵活，产线重组、项目建立随时变化（高频）

**3. 数据权限需求**
- 多工厂数据隔离：工厂A的计划员看不到工厂B的订单
- 权限应基于业务组织，而非行政组织
- 避免行政调整影响业务权限

**4. 与外部系统集成**
- 行政组织：与 HR、OA、ERP 对接，保持一致
- 业务组织：与 PLM、DNC、SCADA 对接，体现制造语义

### 2.5 工作中心（WorkCenter）

**定义**：制造资源的组织单位，用于工艺路线、任务分配、产能计算的工艺资源载体。

**核心定位**：工作中心代表**生产执行资源**，是任务派工的实际承载对象。

#### 工作中心 vs 业务组织

工作中心不是业务组织的一种类型，而是**归属于**业务组织的资源实体。

```
BizOrg (业务组织)  ──── 1:N ────  WorkCenter (工作中心)
     │                                │
     │                                │
管理组织单位                      生产执行资源
数据权限载体                      任务派工载体
```

#### 职责边界

| 对比维度 | 工作中心（WorkCenter） | 业务组织（BizOrg） |
|---------|----------------------|------------------|
| **本质定位** | 工艺资源单位（生产要素） | 管理组织单位（权责主体） |
| **业务语义** | Where（在哪里做） | Who（谁负责管理） |
| **核心用途** | 工艺路线、任务分配、产能计算 | 数据权限、人员归属、成本核算 |
| **关联对象** | 设备、工装、能力参数 | 用户、工作中心、行政组织（可选） |
| **变更频率** | 频繁（工艺调整、产线重组） | 相对稳定（组织架构调整） |
| **权限载体** | 否（不是权限主体） | 是（RBAC 数据范围） |
| **人员关系** | 技能匹配、资源配置 | 组织归属、权限范围 |

#### 工作中心类型与任务派工

**核心理念**：在任务派工场景中，无论是"分派给班组"还是"分派给工作中心"，**本质上都是分派给工作中心**（生产执行资源）。

**工作中心类型枚举**：

| 类型 | 代码 | 说明 | 作为派工对象的语义 |
|------|-----|------|------------------|
| 产线 | CK | 生产线 | 任务分派给产线（如装配线、涂装线） |
| 设备 | SB | 单台设备 | 任务分派给设备（如数控机床、检测设备） |
| **人员** | **RY** | 人工工作中心 | **任务分派给个人**（如手工装配） |
| **小组** | **XZ** | 作业小组 | **任务分派给班组**（如装配小组、检验小组） |
| 外委 | WW | 外协单位 | 任务分派给外协供应商 |

**关键洞察**：

1. **班组即小组类工作中心**：
   - 业务组织中的"班组"（Team）通常对应一个"小组类型"（XZ）的工作中心
   - "分派给班组"实质是分派给小组类型的工作中心
   - 示例：数控工段-白班班组（业务组织）→ 数控白班组（XZ类工作中心）

2. **统一的派工模型**：
   - 所有任务派工都指向 `WorkCenter`（生产执行资源）
   - 工作中心类型决定资源形态（产线/设备/人员/小组/外委）
   - 业务组织字段表示**管理归属**（权限、成本、统计）

3. **任务的双重归属**：

| 字段 | 类型 | 业务含义 |
|------|------|---------|
| `workCenter` | WorkCenter | 执行资源（哪个资源执行任务） |
| `bizOrg` | BizOrg | 管理归属（权限、成本、统计） |
| `factory` | BizOrg | 工厂归属（多工厂隔离） |

#### 工作中心与用户的关系

工作中心通过 `WorkCenterUserLink` 关联用户，业务语义为**技能配置**（哪些人能操作这个工作中心），与 `BizOrgUser`（组织归属，用于数据权限）形成互补。

#### 为什么不将工作中心作为业务组织类型

**核心理由**：

1. **语义差异**：业务组织是"谁负责管理"（权责主体），工作中心是"在哪里做"（资源要素）
2. **权限错位**：数据权限应基于业务组织（管理视角），而非工作中心（技能视角）
3. **关系灵活**：工作中心可能跨组织共享（如共享检测设备），业务组织是严格树形层级
4. **行业实践**：SAP MES、Siemens Camstar 均采用工作中心**归属于**组织，而非"是"组织的一种

**实体模型**：`WorkCenter`
- 表名：`MOM_WORK_CENTER`
- 主要字段：`code`、`name`、`workCenterType`（类型）、`workCenterCategory`（分类）、`factory`（所属工厂）、`timePrecision`（工时精度）

### 2.6 用户与组织的关系

#### 用户 ↔ 行政组织：N:M（多对多）

**关系表**：`OrgUser`
- 表名：`SYS_ORG_USER`
- 关键字段：`orgId`、`userId`

**业务含义**：
- 用户的行政隶属关系（从 HR 系统同步）
- 一个用户可以兼职多个部门
- 一个部门有多个用户

**典型场景**：
- 张三主要在生产部，兼任质量部质量工程师
- 李四从研发部调到生产部（通过时间维度记录历史）

---

#### 用户 ↔ 业务组织：N:M（多对多）

**关系表**：`BizOrgUser`
- 表名：`SYS_BIZ_ORG_USER`
- 关键字段：`bizOrgId`、`userId`

**业务含义**：
- 用户在业务组织中的工作分配
- 一个用户可以在多个车间工作
- 一个车间有多个操作人员
- **用于数据权限计算**（用户所在的业务组织决定其数据权限范围）

**典型场景**：
- 王五主要在机加车间1工作，也可以支援机加车间2
- 赵六是多能工，可以在装配车间和调试车间工作
- 临时工小李被分配到包装车间工作（即使账号被禁用登录）

---

#### 用户 ↔ 工作中心：N:M（多对多）

**关系表**：`WorkCenterUserLink`
- 表名：`MOM_WORK_CENTER_USER_LINK`
- 关键字段：`workCenterId`、`userId`（通过GenericLink继承关系）

**业务含义**：
- 用户的技能配置（哪些人能操作哪些工作中心）
- 一个用户可以操作多个工作中心
- 一个工作中心可以被多个用户操作
- **用于任务分配推荐**（系统推荐能操作该工作中心的用户）

**典型场景**：
- 张三持有数控操作证，可以操作"数控车床01"、"数控车床02"
- "装配线A"可以由王五、赵六、小李操作
- 外协检验员老刘被配置到"外协检验工作中心"

---

#### 三种用户关系对比

| 对比维度 | OrgUser | BizOrgUser | WorkCenterUserLink |
|---------|---------|-----------|-------------------|
| **关系对象** | 用户 ↔ 行政组织 | 用户 ↔ 业务组织 | 用户 ↔ 工作中心 |
| **业务语义** | 行政隶属关系 | 工作分配关系 | 技能配置关系 |
| **数据来源** | HR 系统同步 | MOM 系统维护 | MOM 系统维护 |
| **用于权限** | 否（行政组织不参与业务权限） | **是**（用于计算用户数据权限） | 否（不是权限载体） |
| **用于任务分配** | 否 | 是（组织维度筛选） | **是**（技能维度筛选） |
| **变更频率** | 低（跟随人事调动） | 中（跟随工作安排） | 中高（跟随技能培训、资源调整） |

**使用场景示例**：

**场景：系统推荐任务执行人**
1. **输入**：生产任务，指定执行工作中心="数控车床01"，执行业务组织="机加车间1"
2. **逻辑**：
   - 从 `WorkCenterUserLink` 查询能操作"数控车床01"的用户 → [张三, 李四, 王五]
   - 从 `BizOrgUser` 查询"机加车间1"的用户 → [张三, 李四, 赵六]
   - 取交集 → [张三, 李四]（既在车间1工作，又能操作数控车床）
3. **结果**：推荐张三、李四

**场景：计算用户数据权限**

## 3. 数据模型设计规范

### 3.1 字段设计标准

在实体模型中，涉及"人"的字段，**统一使用 `User`**。

#### 统一原则

所有字段（审计字段、业务角色字段、系统操作字段）都使用 `User` 类型。

**字段类型示例**：

| 字段用途 | 字段类型 | 典型示例 |
|---------|---------|---------|
| 审计字段 | User | `creator`（创建人）、`modifier`（修改人） |
| 业务角色字段 | User | `planner`（计划员）、`operator`（操作工）、`inspector`（检验员）、`reporter`（报工人） |
| 系统操作字段 | User | `approver`（审批人）、`releaser`（下达人）、`closer`（关闭人） |

**命名约定**：
- 优先使用对象引用（如 `planner: User`），不使用 ID 后缀（如 ~~`plannerId: Long`~~）
- 建模工具会自动识别属性类型（系统内置 vs 业务引用）

#### 临时工、外协人员处理方式

- 创建正常的 `User` 记录，`state` = ACTIVE（正常）
- `employedStatus` 可设为 EMPLOYED（在职）
- **通过角色和功能权限控制**访问范围和操作权限
  - 临时工：授予"临时工"角色，仅能报工、查看任务等基本操作
  - 外协人员：授予"外协人员"角色，限定访问范围
  - 第三方检验员：授予"检验员"角色，仅能录入检验结果
- 班长或管理员可代为录入数据，审计字段记录操作人（班长），业务字段记录执行人（临时工）

## 4. 数据权限设计

### 4.1 权限模型

数据权限基于**业务组织（BizOrg）**授予，不基于行政组织（Org）。

#### 基本原则

- **权限授予角色**：数据权限配置在角色上（`RoleBizOrgPermission`），而非直接授予用户
- **基于业务组织**：权限范围通过业务组织定义（工厂、车间、产线）
- **自动过滤**：用户查询数据时，系统自动根据其角色的组织权限过滤结果

#### 权限范围类型

| 范围类型（scope） | 说明 | 典型场景 |
|-----------------|------|---------|
| **ALL（全部）** | 不限制业务组织，拥有所有数据权限 | 集团高层、系统管理员 |
| **ORG（本部门）** | 仅用户所在的业务组织数据 | 车间主任（看本车间）、产线长（看本产线） |
| **ORG_BELOW（本部门及以下）** | 用户所在业务组织及其下级组织数据 | 工厂长（看本工厂及所有车间）、事业部长（看多个车间） |
| **CUSTOMIZE（自定义）** | 指定具体的业务组织列表 | 跨工厂协调员（明确指定工厂A、工厂B） |

#### 角色权限配置

角色的业务组织权限通过 `RoleBizOrgPermission` 实体配置：
- **roleId**：角色ID
- **scope**：权限范围（ALL / ORG / ORG_BELOW / CUSTOMIZE）
- **bizOrgIds**：业务组织ID列表（逗号分隔，仅 CUSTOMIZE 时使用）

---

### 4.2 用户组织权限获取

#### 核心逻辑链路

用户获取业务组织权限的完整路径：

```
用户（User）
  ↓
关联角色（Role）
  ↓
角色权限配置（RoleBizOrgPermission）
  ↓
根据 scope 类型确定组织：
  ├─ ALL → 全部业务组织
  ├─ ORG → User → BizOrgUser → BizOrg（用户所在的组织）
  ├─ ORG_BELOW → User → BizOrgUser → BizOrg + 其子组织
  └─ CUSTOMIZE → 使用配置的 bizOrgIds
  ↓
合并所有角色的组织（取并集）
  ↓
最终的组织权限列表（orgIds）
```

#### 关键环节：用户到业务组织的映射

**问题**：用户（User）如何确定"本部门"是哪个业务组织？

**答案**：通过 User → BizOrgUser → BizOrg 的关联链路

```
1. 用户登录 → 获取 userId
2. 查询用户关联的业务组织 → BizOrgUser 表（WHERE userId = xxx）
3. 获得业务组织列表 → BizOrg（用户所在的工厂、车间等）
```

**示例**：
- 用户：张三（User，id=100）
- 组织归属：`BizOrgUser` 表记录张三在"工厂A"和"机加车间1"工作
- 角色配置：车间主任角色，scope=ORG（本部门）
- 结果：张三获得"工厂A"和"机加车间1"的数据权限

#### 权限计算详细规则

**规则1：超级管理员特殊处理**
- 超级管理员（code=SUPER）：直接返回全部业务组织权限，不受角色限制

**规则2：无角色用户**
- 用户没有关联任何角色：返回 `orgIds=[0]`，表示无权限

**规则3：角色权限合并**
- 用户关联多个角色时，合并所有角色的组织权限（取并集）
- 如果**任一角色**的 scope=ALL，则用户拥有全部组织权限

**规则4：各类 scope 的计算逻辑**

| scope | 计算逻辑 | 依赖关系 |
|-------|---------|---------|
| ALL | 返回全部业务组织（空列表表示全部） | 无 |
| ORG | 通过 `BizOrgUser` 查询用户关联的业务组织 | User → BizOrgUser |
| ORG_BELOW | 先获取用户关联的BizOrg，再获取这些组织的所有子组织 | User → BizOrgUser → BizOrg.children |
| CUSTOMIZE | 直接使用配置的 `bizOrgIds`（逗号分隔） | 无 |

---

### 4.4 Service 层封装

为减少开发复杂度，在 Service 层封装用户-组织转换逻辑。

**常用模式**：
- **查询我的任务**：`queryMyTasks()` → 自动获取当前用户ID → 查询 `WHERE assignedUser = userId`
- **分配任务**：`assignTask(taskId, targetUser)` → 业务字段记录 `assignedUser`，审计字段自动记录 `modifier`

**缓存优化**：用户-组织映射关系是高频查询，应缓存（建议1小时过期或用户登出清除）

## 6. 常见问题解答

### Q1：临时工为什么要创建 User？

**问题**：临时工不需要完整的系统权限，为什么要创建 User 记录？

**回答**：
- **业务追溯需要**：工时统计、绩效考核、质量追溯需要明确记录"谁干的"
- **数据一致性**：所有业务字段统一使用 User，避免数据模型分裂
- **管理便利性**：临时工转正后，直接调整角色权限即可，无需数据迁移
- **权限控制灵活**：通过角色授予有限权限（如仅能报工、查看任务），既满足业务需要又控制风险
- **正确做法**：创建 User，`state` = ACTIVE（正常），通过"临时工"角色限制访问范围

---

### Q2：为什么需要两套组织体系？

**问题**：维护两套组织（行政组织、业务组织）是否过于复杂？

**回答**：不是过度设计，而是制造业现实需求。

- **组织结构差异**：行政按职能划分（对接HR），业务按工艺划分（支撑生产）
- **变更频率不同**：行政组织稳定（低频），业务组织灵活（高频）
- **数据权限需求**：多工厂数据隔离必须基于业务组织，而非行政组织
- **外部系统集成**：行政组织与 HR/ERP 对接，业务组织与 PLM/SCADA 对接

**详见**：2.4节"两类组织的关系 - 为什么需要双组织体系"

---

### Q3：数据权限如何实现？

**问题**：如何控制工厂A的用户看不到工厂B的数据？

**回答**：
1. **权限授予角色**：为角色配置 `RoleBizOrgPermission`（scope + bizOrgIds）
2. **用户到组织的映射**：
   - scope=ORG/ORG_BELOW：通过 User → BizOrgUser 获取用户所在组织
   - scope=CUSTOMIZE：直接使用配置的 bizOrgIds
3. **权限合并**：用户关联多个角色时，取所有角色的组织权限并集
4. **自动过滤**：查询时系统自动添加 `WHERE bizOrg IN (orgIds)`

**关键**："本部门"权限依赖 `BizOrgUser` 表，记录用户在哪些业务组织工作

**详见**：4.2节"用户组织权限获取"

---

### Q4：工作中心与业务组织是什么关系？

**问题**：工作中心也有层级结构，为什么不作为业务组织的一种类型？

**回答**：
- **本质差异**：
  - 工作中心：Where（在哪里做）→ 生产执行资源
  - 业务组织：Who（谁负责管理）→ 管理组织单位
- **关系模式**：工作中心**归属于**业务组织（N:1），而非"是"业务组织的一种
- **典型应用**：
  - 工作中心用于任务派工、工艺路线、产能计算
  - 业务组织用于数据权限、用户归属、成本核算

**分派班组 vs 分派工作中心**：
- **本质统一**：都是分派给工作中心（生产执行资源）
- **班组即小组类工作中心**：业务组织中的"班组"（Team）对应"小组类型"（XZ）的工作中心

**详见**：2.5节"工作中心（WorkCenter）"

---

### Q5：外协人员如何处理？

**问题**：外协公司派人来厂工作，如何在系统中建模？

**回答**：
- **创建 User**：维护外协人员基本信息
- **状态设置**：`state` = ACTIVE（正常），`employedStatus` = EMPLOYED（在职）
- **角色授予**：授予"外协人员"角色，通过角色权限限制访问范围
- **组织归属**：
  - 行政组织：不关联或关联"外协"虚拟组织
  - 业务组织：临时关联工作区域（通过 BizOrgUser）
- **数据记录**：业务字段用 User，可以由外协人员自己操作（受角色权限限制），也可由内部员工代录

---

### Q6：行政组织在系统中有什么作用？

**问题**：行政组织不参与业务逻辑，为什么还要维护？

**回答**：
- **用户归属**：记录用户的行政隶属关系（从 HR 同步）
- **财务核算**：成本按行政组织统计（对接 ERP）
- **审批流程**：领导审批按行政组织层级
- **合规审计**：审计报表需要行政组织视图
- **不是"无用"**：只是"不直接参与制造业务逻辑"

## 7. 总结

### 设计价值

- ✅ **模型简化**：用户即人员，概念统一，易于理解
- ✅ **业务灵活**：支持临时工、外协、第三方等多种用工形式
- ✅ **审计合规**：满足 ISO 27001、等保三级、军工涉密要求
- ✅ **权限清晰**：基于业务组织的多工厂数据隔离机制
- ✅ **架构稳定**：底层设计正确，数据模型不需要迁移

### 关键原则（记住这五句话）

```
【原则1】所有字段统一使用 User（审计字段、业务角色字段、系统操作字段）
【原则2】User 既是账号也是人员，通过角色和功能权限控制访问范围
【原则3】数据权限授予角色，基于业务组织（BizOrg），不基于行政组织（Org）
【原则4】工作中心是生产执行资源，归属于业务组织（N:1），不是业务组织的一种类型
【原则5】优先使用对象引用（planner: User），避免直接使用 ID（plannerId: Long）
```
