# 产品版本管理与发布设计方案

## 一、背景与问题分析

### 1.1 当前现状

**产品概述**：KM-MOM 是一套面向制造企业的生产管理系统，包含 MES（制造执行）、WMS（仓储物流）、APS（计划排程）等多个业务模块，支持私有化部署到客户现场。

**发布节奏**：季度发布正式版本，双周发布补丁版本，采用[语义化版本号](http://192.168.30.77/kmmom-3.0/km-mom-docs/-/blob/58ba1996a11694a55aaba335353db4a99f8b9adb/06-governance/standards/version-strategy.md)（如 v3.4.0）。

**版本发布涉及的内容**：

| 类别         | 说明        | 当前管理方式             |
| ---------- | --------- | ------------------ |
| 应用程序       | 后端服务、前端界面 | ✅ 代码仓库，有版本控制       |
| 数据库脚本-建表语句 | 表结构、初始化数据 | ✅ 低代码平台管理，有版本控制    |
| 数据库脚本-数据修订 | 数据修复、补丁脚本 | ✅ MOM后端GIT仓库，有版本控制 |
| 业务配置       | 系统参数、业务规则 | ✅ MOM系统功能管理，有版本控制  |
| 运行配置       | 服务连接参数等   | ⚠️ 直接在配置中心修改，无历史记录 |
| 安装包        | 可部署的程序文件  | ⚠️ 手动构建，分散存储，无统一管理 |

**已有的自动化能力**：

* 代码提交后自动记录变更日志

* 发版时自动为代码打版本标签

### 1.2 核心痛点

**痛点一：配置分散在多个系统，缺乏统一关联**

```plaintext
现状：
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  GitLab     │    │  低代码平台  │    │  MOM系统    │    │  Nacos      │
│  代码仓库   │    │             │    │             │    │  控制台     │
├─────────────┤    ├─────────────┤    ├─────────────┤    ├─────────────┤
│ ✅ 后端代码  │    │ ✅ 建表脚本  │    │ ✅ 业务配置  │    │ ⚠️ 服务配置  │
│ ✅ 前端代码  │    │   (有版本)  │    │   (有版本)  │    │             │
│ ✅ 数据修订  │    │             │    │             │    │             │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
      │                  │                  │                  │
      └──────────────────┴──────────────────┴──────────────────┘
                              ↓
              各系统独立管理版本，缺乏统一的版本关联
              发布时需要人工协调多个系统的版本对应关系
```

* 应用程序代码、数据修订脚本在代码仓库中管理

* 建表脚本在低代码平台中管理，有独立版本

* 业务配置在MOM系统中管理，有独立版本

* 运行配置直接在配置中心修改，无法追溯历史变更

* **核心问题**：各系统版本独立，发布时需人工确认版本对应关系，缺乏统一的版本关联机制

**痛点二：安装包缺乏统一管理**

```plaintext
场景：项目 A 需要部署 v3.4.0 版本

当前流程：
  项目经理 → 联系测试人员 → 测试人员查找/重新构建 → 通过网盘/邮件发送
                ↓
        ❌ 效率低、易出错、无法追溯
```

* 安装包由测试人员手动构建，缺乏标准化流程

* 安装包存储分散（本地、共享盘、网盘），无统一存储位置

* 安装包与版本号关联不清晰，难以确认对应的代码版本

## 二、目标与原则

### 2.1 核心目标

```plaintext
一个版本号 = 一个完整的、可部署的、可追溯的发布包
```

具体目标：

1. **完整性**：一个版本包含所有组件（后端、前端、数据库、配置、中间件配置）

2. **可追溯性**：任何运行环境都能查询到当前部署的精确版本

3. **可升级性**：支持从任意版本平滑升级到目标版本

4. **可回滚性**：升级失败时能够快速回滚

### 2.2 设计原则

| 原则    | 说明                    |
| ----- | --------------------- |
| 单一版本号 | 所有组件使用统一的版本号，如 v3.4.0 |
| 制品不可变 | 构建后的制品不可修改，保证一致性      |
| 配置即代码 | 所有配置纳入 Git 管理         |
| 增量迁移  | 数据库和配置变更采用增量脚本        |
| 自动化优先 | CI/CD 自动构建、打包、发布      |

## 三、整体架构设计

### 3.0 产品依赖关系

MOM 产品依赖低代码平台，两者作为独立产品各自发展：

**设计原则**：

* 低代码平台作为独立产品，有自己的发布包和安装工具

* MOM 只在 POM 中声明 SDK 版本，在 manifest.json 中声明依赖的低代码平台版本

* 部署时分别部署两个产品，MOM 不包含低代码平台的制品

**当前问题**：

1. 低代码平台 SDK 在 MOM 的 POM 中使用 `1.0-SNAPSHOT`，无法追溯具体版本

2. 低代码平台前端代码临时放在 MOM 前端仓库，需要迁移

3. 低代码平台缺乏独立的版本发布机制和安装工具

### 4.1 配置仓库设计（后端仓库管理？）

#### 4.1.1 仓库定位

新建独立的 Git 仓库（km-mom-next-config），用于**聚合各系统配置的版本快照**，包括从低代码平台导出的数据库建表脚本、从MOM系统导出的业务配置、以及Nacos配置，与代码仓库同步打 Tag，确保版本一致性。

**核心作用**：

* 将分散在多个系统的配置统一关联到产品版本号

* 发布时自动从各系统导出配置，形成版本快照

* 通过 Git Tag 实现配置的版本追溯

**不包含**：

* 中间件安装配置（由安装工具负责）

* 安装部署脚本（由安装工具负责）

#### 4.1.2 目录结构

```plaintext
km-mom-next-config/
├── config/                                # 配置包
│   ├── prod_KMMOM_v3.4.0.zip              # 数据库脚本（全量，幂等）
│   ├── product_configs_v3.4.0.zip         # 业务配置
│   └── nacos/                             # Nacos 配置
│       ├── nacos_configs_v3.4.0.zip
│       └── nacos.env                      # 环境变量清单
│
├── manifest.json                          # 版本元数据
└── README.md
```

#### 4.1.3 与安装工具的职责划分

| 职责       | 配置仓库      | 安装工具     |
| -------- | --------- | -------- |
| 数据库脚本    | ✅ 存储和版本管理 | 执行脚本     |
| 业务配置     | ✅ 存储和版本管理 | 导入配置     |
| Nacos 配置 | ✅ 存储完整导出  | 导入配置     |
| 中间件依赖清单  | ✅ 声明依赖    | 检查/安装中间件 |
| 中间件安装配置  | ❌         | ✅ 负责     |
| 部署脚本     | ❌         | ✅ 负责     |

#### 4.1.4 配置文件版本号规范

**现有问题**：配置文件版本号与产品版本号不一致（如 `prod_KMMOM_v1.9.2.zip`、`product_configs_v1.0.13_xxx.zip`）

**规范要求**：所有配置文件版本号必须与 Git Tag 保持一致

| 文件类型     | 命名规范                            | 示例                           |
| -------- | ------------------------------- | ---------------------------- |
| 数据库全量脚本  | `prod_KMMOM_{version}.zip`      | `prod_KMMOM_v3.4.0.zip`      |
| 业务配置     | `product_configs_{version}.zip` | `product_configs_v3.4.0.zip` |
| Nacos 配置 | `nacos_configs_{version}.zip`   | `nacos_configs_v3.4.0.zip`   |

#### 4.1.5 设计优缺点

**设计优点**：

* 自包含：每个版本独立完整，通过 Tag 即可获取所有内容

* 幂等设计：database/business-config/nacos 均可重复执行

* 升级简单：直接使用目标版本的配置包，无需考虑升级路径

* 职责分离：配置仓库只管"是什么"，安装工具负责"怎么装"

### 4.2 版本号嵌入方案

**目标**：让运行中的系统能够回答"我是什么版本"

#### 4.2.1 后端 JAR 包版本嵌入

**方案**：Maven 构建时注入版本信息到 MANIFEST.MF

**实现要点**：

* 使用 maven-jar-plugin 写入 Build-Version、Build-Timestamp

* 使用 git-commit-id-plugin 获取 Git Commit 信息

* 提供 `/api/version` 端点供查询单服务版本

* 网关层提供 `/api/system/versions` 聚合查询所有服务版本

#### 4.2.2 前端 dist 包版本嵌入

**方案**：构建时生成 version.json 文件

**实现要点**：

* CI 构建时通过环境变量传入版本号

* 构建脚本生成 `dist/version.json`，包含 version、buildTime、gitCommit

#### 4.2.3 数据库版本管理

**方案**：沿用 `DEV_RUNTIME_PRODUCT` 表记录产品版本

**实现要点**：

* 沿用现有表结构，使用 `CODE`（产品代码）、`VERSION`（版本号）、`INSTALL_TIME`（安装时间）字段

* 安装工具执行脚本完成后写入版本记录

* km-mom-platform 提供 `/api/version/database` 查询接口

#### 4.2.4 关于页面展示

版本一致时显示统一产品版本，不一致时分别显示各组件版本：（永远显示所有版本）

```plaintext
┌─────────────────────────────────────────┐
│           系统版本信息                   │
├─────────────────────────────────────────┤
│  产品版本      │  v3.4.0                │
│  构建时间      │  2025-01-15 10:30:00   │
└─────────────────────────────────────────┘
```

### 4.3 CI/CD 流水线设计

版本发布涉及两个流水线：

#### 4.3.1 版本发布流水线（打 Tag）

**现有能力**：

* 参数化构建：输入版本号和版本说明

* 多仓库处理：自动遍历后端、前端、配置仓库

* 生成变更日志：使用 Claude Code 智能解析 commits

* 自动打 Tag 并创建 GitLab Release

**配置仓库特殊处理**：

#### 4.3.2 Nacos 配置环境无关化规范

**问题**：Nacos 配置中硬编码 IP、密码等环境特定信息，导致配置无法跨环境使用

**解决方案**：使用环境变量占位符

```yaml
# 示例：环境无关的配置
spring:
  datasource:
    url: jdbc:mysql://${MYSQL_HOST:localhost}:${MYSQL_PORT:3306}/${MYSQL_DATABASE:km_mom}
    username: ${MYSQL_USERNAME:root}
    password: ${MYSQL_PASSWORD:123456}
```

**规范要求**：

| 配置类型     | 环境变量格式                    |
| -------- | ------------------------- |
| 数据库地址    | `${MYSQL_HOST:默认值}`       |
| Redis 连接 | `${REDIS_HOST:默认值}`       |
| 密码类      | `${XXX_PASSWORD:}`        |
| 外部服务地址   | `${SERVICE_ENDPOINT:默认值}` |

**禁止出现**：硬编码的 IP 地址、密码、非标准端口

**环境变量清单**：后端仓库 `2_deploy/nacos.env` 文件，记录所有环境变量及默认值，开发人员可直接 `source` 使用

#### 4.3.3 制品构建流水线

**触发方式**：版本发布流水线完成后自动触发，或手动触发

**流程**：

1. 并行检出三个仓库的 Tag 代码

2. 并行构建后端 JAR 和前端 dist

3. 组装发布包（复制配置、生成 manifest.json、生成 checksums）

4. 上传到 Nextcloud 制品仓库

### 4.4 部署与升级方案（删除）

#### 4.4.1 部署与升级场景

| 场景   | 定义           | 典型案例                            |
| ---- | ------------ | ------------------------------- |
| 全新安装 | 目标环境无历史数据    | 新项目启动、新客户交付                     |
| 版本升级 | 从任意版本升级到目标版本 | v3.2.0 → v3.4.0、v3.4.0 → v3.4.1 |

#### 4.4.2 升级策略

**直接升级原则**：

* 支持从任意版本直接升级到目标版本，无需逐版本升级

* 原因：所有配置包（数据库脚本、业务配置、Nacos配置）均为全量且幂等设计

**幂等执行原则**：

* 所有脚本和配置包支持重复执行，数据库脚本改造指南详见 数据库脚本编写规范

* 升级中断后可从当前步骤重新执行

#### 4.4.3 升级流程

**升级步骤**：

1. 执行 config/prod\_KMMOM\_\*.zip（全量，幂等）

2. 执行 config/product\_configs\_\*.zip（幂等）

3. 导入 config/nacos/\*.zip（幂等）

4. 更新应用制品（后端 JAR、前端 dist）

5. 验证版本

### 4.5 外部依赖管理

#### 4.5.1 背景

MOM 产品依赖低代码平台，低代码平台包含以下组件：

| 组件                 | 类型      | 说明                  |
| ------------------ | ------- | ------------------- |
| km-dev-sdk         | SDK JAR | MOM 编译时依赖，通过 POM 引入 |
| km-dev-runtime     | 后端微服务   | 低代码平台运行态后端          |
| km-dev-design-web  | 前端      | 低代码设计器前端            |
| km-dev-runtime-web | 前端      | 低代码运行态前端            |

#### 4.5.2 设计原则

**低代码平台作为独立产品，与 MOM 分开部署**：

**命令行**

**MOM 侧只需要**：

1. POM 中声明 SDK 的正式版本号（非 SNAPSHOT）

2. manifest.json 中声明依赖的低代码平台最低版本

3. 部署文档中说明需要先部署低代码平台

**MOM 发布包不包含低代码平台的制品**，由低代码平台团队提供独立的发布包和安装工具。

#### 4.5.3 manifest.json 依赖声明

```json
{
  "version": "3.4.0",
  "dependencies": {
    "dev-platform": {
      "version": "2.1.0",
      "sdkVersion": "2.1.0",
      "description": "低代码平台，需单独部署，请联系低代码平台团队获取安装包"
    }
  }
}
```

#### 4.5.4 版本兼容性矩阵

| MOM 版本 | 低代码平台版本 |
| ------ | ------- |
| v3.4.0 | 2.1.0   |
| v3.3.0 | 2.0.0   |
| v3.2.0 | 1.9.0   |

## 七、总结

本方案通过以下核心机制解决产品版本管理的五大痛点：

| 痛点               | 解决机制                                      |
| ---------------- | ----------------------------------------- |
| 配置分散在多个系统，缺乏统一关联 | 配置仓库聚合各系统配置快照 + 统一 Tag                    |
| 安装包缺乏统一管理        | CI/CD 自动构建 + Nextcloud 存储                 |
| 运行环境版本无法确认       | 版本嵌入 + 关于页面 + 版本 API                      |
| 历史版本无法重建         | 配置仓库 + Tag 回溯                             |
| 版本升级             | manifest.json + 全量幂等配置包（支持直接升级到任意版本）      |
| 外部依赖版本不明确        | POM 版本锁定 + manifest.json 声明依赖 + 低代码平台独立发布 |

实施后，**一个版本号（如 v3.4.0）将能够关联并获取到所有组件的对应制品**，形成一个完整的、可部署的、可追溯的、可升级的发布包。
