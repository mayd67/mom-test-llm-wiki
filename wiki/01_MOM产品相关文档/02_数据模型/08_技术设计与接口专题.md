# 技术设计与接口专题

## 一、WebAPI 规范与接口清单

来源：

- `webapi/webapi-specification.md`
- `webapi/webapi-list.md`

### 1. 设计重点

- 对外公开 API 必须放在所属模块的 `remote/api` 包下
- API 文档基于 `@Tag`、`@Operation`、`@Schema` 等注解自动生成
- URL 路径遵循版本化和资源化规则
- 当前接口清单共 9 个接口，主要分布在：
  - `MES` 生产订单相关接口
  - `Approval` 审批流程相关接口

### 2. 测试关注点

- 接口路径、分组、版本号是否符合规范
- DTO/VO 字段说明是否完整
- 请求示例、响应示例、错误返回是否一致
- API 文档与真实实现是否一致
- 网关聚合后的文档是否能正确展示

## 二、WebHook 规范

来源：`tech-design/webhook-specification.md`

### 1. 设计重点

- WebHook 采用 `domain/category/code/name/description` 五要素
- 编码统一格式：`mom.{domain}.{category}.{action}.{timing}`
- 触发位置优先放在应用服务层
- 消息结构、订阅消费原则和命名规范有明确约束

### 2. 测试关注点

- WebHook 编码、命名、描述是否符合规范
- before/after 时机是否按设计触发
- 应用服务层是否真的发出了事件
- 订阅方拿到的消息结构是否完整

## 三、元数据与数据模型使用

来源：`tech-design/metadata-usage-guide.md`

### 1. 设计重点

- 产品包、应用包、业务配置包三种交付物分离
- 平台通过标准化制品包承接元数据发布、安装、消费
- 强调元数据驱动、安装、缓存和消费链路
- 数据权限、工厂隔离、业务审计等能力通过拦截器织入

### 2. 测试关注点

- 元数据包安装后是否完整生效
- 预定义数据、业务配置和结构变更是否按包类型分离
- 数据权限、工厂隔离、业务日志是否在元数据驱动下稳定执行
- 元数据缓存更新、查询和高并发访问是否正确

## 四、版本管理与发布

来源：`tech-design/version-release-design.md`

### 1. 设计重点

- 版本发布涉及应用程序、建表脚本、数据修订、业务配置、运行配置、安装包
- 目标是建立统一版本关联关系
- 引入 manifest、版本嵌入、配置仓库、CI/CD 流水线等设计

### 2. 测试关注点

- 当前运行版本是否可追溯
- 应用、数据库、配置、安装包是否属于同一发布版本
- 升级、回滚、兼容性矩阵是否清晰
- 发布说明、安装包与版本标签是否一致

## 五、用户组织与消息体系

来源：

- `tech-design/user-person-org-design.md`
- `tech-design/system-message-design.md`

### 1. 为什么这里也要看

这两份文档虽然已在平台专题中整理，但在技术层也有明显约束：

- 数据权限和组织范围会影响接口与查询结果
- 错误码和统一响应结构会影响接口测试与自动化校验

### 2. 测试关注点

- 接口返回是否体现组织隔离
- 错误码、提示语、成功响应结构是否统一
- 业务错误与技术异常是否被正确区分

## 六、后续生成测试点时怎么用

当需求涉及以下场景时，优先查本专题：

- 对外 API
- 审批流或生产订单公开接口
- WebHook 回调
- 元数据包安装与版本发布
- 错误码、消息提示、统一响应格式

建议生成测试点时至少覆盖：

- 协议结构
- 版本和路径规范
- 错误返回
- 权限和组织范围
- 事件触发与消费
