# 原始文档导航

本页用于给测试工程师提供一个统一入口，直接在 `wiki` 中访问 `03-development` 的原始文档镜像。

当前策略：

- 原文已经较规整，优先保留原结构和原上下文
- `wiki` 主要承担导航职责，不再过度拆散原文
- 后续生成测试点和测试用例时，优先从这里进入原始设计资料

## 一、原文镜像入口

原文镜像目录：

`wiki/01_MOM产品相关文档/02_数据模型/02_原始文档/03-development/`

其下完整保留四类资料：

- `datamodel-design/`：数据模型设计
- `module-design/`：模块设计
- `tech-design/`：技术设计
- `webapi/`：接口规范与接口清单

## 二、按用途导航

### 1. 写测试点时优先看

- `module-design/`
- `datamodel-design/`

适用问题：

- 这个需求要测哪些业务流程
- 这个功能改了哪些对象、字段、状态和联动链路
- 这个需求的边界、兼容、异常和回归范围是什么

### 2. 做数据与状态分析时优先看

- `datamodel-design/`

适用问题：

- 主单和明细怎么关联
- 状态字段和枚举怎么定义
- 哪些字段是回写字段、冗余字段、默认字段

### 3. 做接口与联动测试时优先看

- `webapi/`
- `tech-design/`

适用问题：

- 对外接口路径、请求结构、响应结构是什么
- 错误码、消息提示、统一返回结构是什么
- WebHook、元数据、版本与发布相关测试依据在哪里

## 三、四类原文的测试价值

### 1. datamodel-design

重点价值：

- 看对象、字段、关系、状态、枚举、初始化依赖

### 2. module-design

重点价值：

- 看需求实现边界、主流程、接口设计、数据流转、风险与待确认项

### 3. tech-design

重点价值：

- 看平台级规则、权限模型、消息反馈、版本、元数据、WebHook 约束

### 4. webapi

重点价值：

- 看 API 规范、公开接口清单、参数结构、返回结构

## 四、推荐使用方式

建议顺序：

1. 先看需求文档
2. 再通过本页进入对应的 `module-design/`
3. 如涉及对象和状态，再看 `datamodel-design/`
4. 如涉及接口、错误码、对外能力，再看 `webapi/` 或 `tech-design/`

## 五、与专题页的关系

当前 `02_数据模型/` 下已有专题页和总览页，它们仍然保留，但定位调整为：

- 专题页：快速建立测试理解
- 原文镜像：作为正式设计依据和主查询入口

当原文已经足够规整时，优先读原文；只有在需要快速总结时，再回看专题页。
